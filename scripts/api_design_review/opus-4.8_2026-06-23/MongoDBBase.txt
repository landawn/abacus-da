1. OVERALL ASSESSMENT
MongoDBBase is the shared base for both the blocking and reactive MongoDB facades and holds the static
BSON/extract/stream helper surface. It is broadly healthy: the fromJson/toJson, toList/toEntity/readRow,
extractData (x6), and stream (x4) families have consistent return conventions (mutable List, abacus
Stream). The actionable items are an unsafe Object-vs-varargs overload family and a few doc/parity nits.
[RE-REVIEW 2026-06-24: lines drifted ~+1/+2 from the original report (file is now 1776 lines). All four
 Object-vs-varargs overload pairs STILL EXIST (§3 STILL-VALID) but a key sub-claim — that the varargs
 pair-parser silently accepts non-String names — is now FIXED: toDocument/toBSONObject/toDBObject eagerly
 throw IAE on non-String names (MongoDBBase.java:610-613,673-676,778-781). toMap is STILL shallow
 (STILL-VALID, MongoDBBase.java:902-907). objectId2Filter $in companion STILL MISSING (STILL-VALID).
 registerIdProperty still @Deprecated (KEEP). The OTHER/doc objectId2Filter fix is confirmed present
 (MongoDBBase.java:222-223,254-255). Current line citations below.]

3. RETURN-TYPE AND PARAMETER ISSUES
- toBson / toDocument / toBSONObject / toDBObject each have a (Object obj) AND a (Object... a) overload |
  MongoDBBase.java:449/482, 517/553, 658/720, 763/828 | Medium | Classic unsafe varargs-over-Object: a
  single Object[] argument binds to the (Object...) "name/value pairs" overload, never the single-value
  overload — and the two have incompatible meaning (one entity/map vs alternating key/value pairs).
  Verified all four pairs exist. Recommend: give the pairs form a distinct, explicit name (e.g.
  toBsonFromPairs(Object...)) or require a Map, so the dispatch can't be silently wrong.
  [RE-REVIEW 2026-06-24: STILL-VALID (lines updated). All four (Object)+(Object...) pairs still present:
   toBson 449/482, toDocument 517/553, toBSONObject 658/720, toDBObject 763/828. The fundamental
   overload-ambiguity (a single Object[] always binds to the pairs overload) is unchanged — no rename or
   Map-requirement was applied. PARTIAL HARDENING since original: the pair-parsers now eagerly reject
   non-String names with IAE instead of a raw ClassCastException (toDocument 610-613, toBSONObject 673-676,
   toDBObject 778-781), and toBson(Object...)/toDocument(Object...) explicitly route length==1 through the
   single-value overload and length 0 to an empty doc (482-484 -> 553-558). This narrows the failure modes
   but does NOT remove the design-level dispatch hazard the finding targets; finding stands at Medium.]
- toMap(Document) / toMap(Document, IntFunction) | MongoDBBase.java:866,902 | Low | Shallow conversion —
  nested Documents remain Document instances, though the name implies full materialization. Documented;
  consider a deepToMap sibling. Confirming check: confirm nested values are not recursively converted.
  [RE-REVIEW 2026-06-24: STILL-VALID — confirmed shallow. toMap(Document) delegates to the IntFunction
   overload (MongoDBBase.java:867), which does `map.putAll(doc)` (MongoDBBase.java:905) with NO recursive
   conversion, so nested Document values stay Document instances. The Javadoc now explicitly documents this
   ("The copy is shallow: a nested Document stays a Document", MongoDBBase.java:856-857), so it is
   documented-by-design; the deepToMap sibling suggestion remains an optional enhancement.]

CROSS-CLASS NOTES
- This base is shared by mongodb/MongoDB and reactivestreams/MongoDB (correct reuse, not duplication).
- The Object-vs-varargs ambiguity recurs anywhere the library offers "single value" + "key/value pairs"
  on the same method name; flag as a convention to settle library-wide.

NEW ISSUES (2026-06-24 re-review)
- Asymmetric non-String-name validation across the BSON builders | MongoDBBase.java:610-613 (toDocument),
  673-676 (toBSONObject), 778-781 (toDBObject) vs 553-559 (toBson(Object...)) | Low (consistency nit) |
  The three concrete pair-parsers now eagerly reject non-String names with IAE, which is good, but the
  policy is not uniform with the AnyUtil pair-helper (aws/AnyUtil.java:82) that coerces non-String names
  via N.stringOf. Same observation already logged in the AnyUtil report §3; noted here for completeness as
  a library-wide convention to settle (reject vs coerce). Not a defect in MongoDBBase itself.
