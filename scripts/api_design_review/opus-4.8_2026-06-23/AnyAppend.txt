1. OVERALL ASSESSMENT
AnyAppend is a faithful fluent wrapper over HBase `Append`. No correctness defects. Findings are a
redundant attribute overload and cross-class factory parity. NOTE: an earlier draft flagged add(Cell) as
"silently swallowing" a wrong-row exception — the WRAPPER itself has no try/catch and declares no checked
exception. The precise mechanism (verified by disassembling hbase-client 2.6.0): HBase's Append.add(Cell)
override CATCHES the row-mismatch IOException, calls LOG.error(...), and returns without rethrowing — i.e.
it logs-and-swallows. The wrapper's own Javadoc (AnyAppend.java:583-586) accurately documents exactly this
("logs and swallows the row-mismatch IOException ... callers must enforce row consistency themselves"), so
the doc is correct and was left unchanged. The Put/Delete/Increment-vs-Append difference in the
`throws IOException` clause is a faithful mirror of HBase's own subclass API and is NOT a defect.
[RE-REVIEW 2026-06-24: Re-verified the full file (726 lines) against live source. All findings below
re-confirmed at current line numbers. add(Cell) is unchanged at AnyAppend.java:604 (no throws clause);
its log-and-swallow Javadoc is still at AnyAppend.java:583-586 (exact). The setAttribute(String,byte[])
overload is now documented as an intentional raw-bytes fast path (AnyAppend.java:610-639), shifting that
finding to PARTIALLY-ADDRESSED. No new correctness defects found.]
