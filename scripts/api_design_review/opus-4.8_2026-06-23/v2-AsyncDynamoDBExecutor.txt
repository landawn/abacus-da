Reviewer: opus-4.8  Date: 2026-06-23  (orchestrator-verified against source)

1. OVERALL ASSESSMENT
The v2 async executor wraps the natively-async AWS SDK v2 DynamoDbAsyncClient. Naming, parameter ordering,
overload families, placement, and @Beta/@Deprecated parity are clean. There are two notable items: a
library-wide async-type deviation (CompletableFuture vs ContinuableFuture) and blocking auto-pagination
that the Javadoc under-discloses on two methods.

[RE-REVIEW 2026-06-24] Re-verified against current source. Headline CompletableFuture-vs-ContinuableFuture
finding STILL-VALID (193 CompletableFuture, 0 ContinuableFuture). Blocking auto-pagination on list/query
STILL-VALID (list/query Javadoc still omit the .get()-blocking note that stream/scan carry). Converter
rename in the sync twin (attrValueOf -> toAttributeValue) propagated here too. One NEW issue: query's
null-targetClass handling now diverges from the (tightened) sync twin. See per-finding tags + NEW ISSUES.

3. RETURN-TYPE AND PARAMETER ISSUES
- WHOLE CLASS returns native java.util.concurrent.CompletableFuture, not the library-standard
  ContinuableFuture | v2/AsyncDynamoDBExecutor.java (class-wide; Javadoc :78-167) | Medium (design
  decision) | Verified: 193 CompletableFuture, 0 ContinuableFuture, whereas every other async executor
  (AsyncHBaseExecutor, AsyncMongoCollectionExecutor, AsyncCassandraExecutorBase, AND the v1 dynamo async
  cousin) uses ContinuableFuture. Deliberate and documented (the v2 client is natively non-blocking), but
  it is the single biggest cross-class convention break. DECISION NEEDED (SUMMARY P2): either standardize
  by wrapping to ContinuableFuture, or adopt CompletableFuture library-wide and document it as the
  forward direction.
  [RE-REVIEW 2026-06-24: STILL-VALID — re-counted current source: 193 CompletableFuture, 0
