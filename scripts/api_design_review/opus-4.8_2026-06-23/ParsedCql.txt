API DESIGN REVIEW — cassandra/ParsedCql.java
public final class ParsedCql
Reviewer: opus-4.8  Date: 2026-06-23  (orchestrator-verified against source)

1. OVERALL ASSESSMENT
ParsedCql is the parsed, POOLED/CACHED form of a CQL statement (originalCql / parameterizedCql /
namedParameters / parameterCount), built via the static factory parse(String) and backed by a
KeyedObjectPool. It is the CQL analogue of the core ParsedSql. The design is sound; the one real issue is
that its only collection accessor leaks the live internal map of a shared, pooled instance.

2. NAMING ISSUES
None.

3. RETURN-TYPE AND PARAMETER ISSUES
- namedParameters() returns the live mutable internal Map<Integer,String> | ParsedCql.java:561-565 |
  Medium | Verified: the body is `return namedParameters;` (the internal field), and ParsedCql instances
  are POOLED/CACHED (parse() is backed by KeyedObjectPool). A caller that mutates the returned map
  corrupts the cached instance for every other holder. The Javadoc only "asks" for read-only. Return an
  ImmutableMap / Collections.unmodifiableMap — consistent with CqlMapper.getAttributes() which already
  returns ImmutableMap. (Map<Integer,String> shape itself is an intentional ParsedSql mirror; only the
  mutability is actionable.)

None.

CROSS-CLASS NOTES
- Mutable-internal-collection leak is the same class of issue seen in HBase (AnyMutation.getFamilyCellMap)

- ParsedCql (parser output) + CqlMapper (registry) + CqlBuilder (generator) form a coherent pipeline.
