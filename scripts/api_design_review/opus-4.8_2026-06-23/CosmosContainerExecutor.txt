API DESIGN REVIEW — azure/CosmosContainerExecutor.java
public class CosmosContainerExecutor   (class is @Beta)
Reviewer: opus-4.8  Date: 2026-06-23  (orchestrator-verified against source)

1. OVERALL ASSESSMENT
CosmosContainerExecutor has two halves: a faithful 1:1 wrap of the Azure Cosmos SDK CosmosContainer
(createItem/upsertItem/patchItem/queryItems/readMany/readAllItems — correctly NOT renamed to library
verbs) and a thinner library-integrated query path. The library-integrated path is notably incomplete vs
the other condition-based executors (mongo/cassandra): it exposes only streamItems over a Condition, with
no Optional point-read, no exists/count, and no paged/first-result variants. This is mitigated by the
class being @Beta and intentionally thin, but the gaps are real. id/partitionKey ordering is clean
throughout.

2. NAMING ISSUES
None. (replaceItem deliberately reorders to lead with (id,partitionKey) vs the SDK's (item,id,pk) — a
reasonable library-ergonomics choice.)

3. RETURN-TYPE AND PARAMETER ISSUES
- readItem(String,PartitionKey,[opts],Class) -> CosmosItemResponse<T> | CosmosContainerExecutor.java:864,
  911 | High | No absence-returning point-read: a miss throws CosmosException(404). The library discipline
  is get->Optional / gett->nullable. Add get(id,partitionKey,Class)->Optional<T> wrapping readItem and
  mapping 404 -> empty.
- streamItems(Condition,...) family returns Stream only | :1418-1751 | High | No List/DataSet/Optional
  variants over a Condition (vs mongo's stream/list/query(DataSet)/findFirst quartet). Add at least a
  paged queryItems(Condition)->CosmosPagedIterable and a findFirst(Condition)->Optional. (Both raw-SQL
  surfaces — String and SqlQuerySpec — already have BOTH queryItems and streamItems, so the Condition path
  is the lone asymmetry.)
- deleteItem(T item, options) -> CosmosItemResponse<Object> | :739 | Low | <T> is input-only while the
  response is <Object>; the type param buys nothing. Cosmetic; could drop <T>.

4. MISPLACED METHODS
None.

5. MISSING METHODS
- exists(id,partitionKey) -> boolean and count([Condition]) -> long absent | CosmosContainerExecutor.java
  | High | Verified absent. Both are core library-discipline methods present on every other executor
  (mongo:262/334) and trivially expressible over Cosmos. Add them.
- prepareQuery(...,int count) has a dead `if (count>0) limit(count)` branch (only caller passes 0) |
  :1808,1811,1855 | Medium | The plumbing for a top-N/limit on the Condition path exists but no public
  overload reaches it. Expose a limit/top-N overload, or delete the dead branch.

6. DEPRECATION / REMOVAL CANDIDATES
None.

OTHER (annotation hygiene)
- @Beta granularity | streamAllItems carries method-level @Beta (:1146,1186) but the more-experimental
  Condition streamItems overloads (:1596+) do not (class is already @Beta) | Low | Make consistent.

CROSS-CLASS NOTES
- Biggest deviation: the library-integrated query path is far thinner than mongo/cassandra (no Optional
  point-read, no exists/count, no paged/first for the Condition path). Acceptable while @Beta, but this is
  the class to round out before GA.
