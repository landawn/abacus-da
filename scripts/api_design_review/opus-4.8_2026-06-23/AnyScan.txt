API DESIGN REVIEW — hbase/AnyScan.java
public final class AnyScan extends AnyQuery<AnyScan>
Reviewer: opus-4.8  Date: 2026-06-23  (orchestrator-verified against source)

1. OVERALL ASSESSMENT
AnyScan is a faithful fluent wrapper over HBase `Scan` and is generally consistent with its AnyQuery
sibling AnyGet. All findings are consistency/surface-pruning items — there are no correctness bugs. The
class already carries appropriate @Deprecated markers on its legacy factories. The main themes are: a
pair of redundant overrides that duplicate base behavior, several already-deprecated factories that can
be removed, and a minor accessor-naming asymmetry vs AnyGet.
[RE-REVIEW 2026-06-24: Re-verified all findings against current source. Most line numbers held; two were

Every finding is STILL-VALID in code; the two "Javadoc contradiction" items (getFamilies null semantics,
setFamilyMap "must not be null") are now PARTIALLY-ADDRESSED via added documentation but the underlying
code is unchanged. No new code issues. See per-finding RE-REVIEW tags.]

3. RETURN-TYPE AND PARAMETER ISSUES
[RE-REVIEW 2026-06-24: PARTIALLY-ADDRESSED — getFamilies() at AnyScan.java:509 is still a bare delegate to
scan.getFamilies() returning nullable byte[][]. The recommended documentation was applied: Javadoc now
states "Returns null if no families have been added (in which case the scan covers every family in the
table)" and "@return ... null if no families specified" (AnyScan.java:493-505). Null semantics preserved
(KEEP). The only outstanding item — picking ONE canonical accessor with AnyGet — is the cross-class item.]
- getFamilies() returns nullable byte[][] | AnyScan.java:509 | Low | Returns null when no families
  added. Unlike getFamilyMap() (never-null) this can surprise callers, BUT null here is a MEANINGFUL
  HBase signal ("no family filter -> scan all families"), faithfully mirrored from Scan.getFamilies().
  Recommendation: KEEP the null semantics but document "null = all families" explicitly; do not convert
  to empty array (that would lose the all-families signal). Verified: bare delegate to scan.getFamilies().
[RE-REVIEW 2026-06-24: PARTIALLY-ADDRESSED — setFamilyMap(Map) at AnyScan.java:631 is still a bare delegate
to scan.setFamilyMap(familyMap) with NO null guard. The @param still says "must not be null"
(AnyScan.java:627), so the wording mismatch is not fully resolved; HOWEVER the Javadoc body and a usage
edge-note now openly describe the null-tolerant reality ("A null map is accepted here (stored by
reference); a later traversal would fail", AnyScan.java:619-625). Recommendation now: align the @param
wording with the body (drop "must not be null") OR add the fail-fast guard.]
- setFamilyMap(Map) Javadoc says "must not be null" but performs no null check | AnyScan.java:631 | Low |
  Either add a fail-fast null guard or relax the contract wording. Verified: body is a bare delegate.
