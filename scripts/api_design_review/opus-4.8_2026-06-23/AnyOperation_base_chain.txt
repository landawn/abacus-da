API DESIGN REVIEW — HBase "Any*" base chain
Classes: AnyOperation, AnyOperationWithAttributes, AnyQuery, AnyMutation
(package-private abstract bases; their PUBLIC methods form part of the public API of the
public final subclasses AnyGet/AnyScan/AnyPut/AnyDelete/AnyIncrement/AnyAppend, so they are in scope)
Reviewer: opus-4.8  Date: 2026-06-23  (orchestrator-verified against source)

1. OVERALL ASSESSMENT
The base chain is a faithful, well-documented fluent wrapper over the Apache HBase client class
hierarchy (Operation -> OperationWithAttributes -> Query/Mutation). Self-typed returns (AO/AOWA/AQ/AM)
give correct fluent chaining to every subclass without per-subclass overrides. The getACL/setACL pair
appearing on both AnyQuery and AnyMutation is NOT duplication — HBase declares them separately on Query
and Mutation, so hoisting them up would be wrong. Deprecations (compareTo(Row) -> Row.COMPARATOR) are
applied consistently. The only actionable items are a checked-exception policy inconsistency and two
minor mutable/immutable + naming nits; there are no misplaced or missing methods.

2. NAMING ISSUES
- get(String,String) / get(byte[],byte[]) returning List<Cell> | AnyMutation.java:439,453 | Low |
  A method named `get` returning List<Cell> reads ambiguously in a library that otherwise leans on
  Optional/Stream. Faithful mirror of HBase Mutation.get; recommend KEEP but consider an alias
  `cells(family,qualifier)`. Confirming check: confirm no sibling uses a `cells()`/`getCellList()` name
  that this should match (none found in the Any* family).

3. RETURN-TYPE AND PARAMETER ISSUES
- Checked-exception policy inconsistency: toJson()/toJson(int) wrap HBase IOException as
  UncheckedIOException (AnyOperation.java:128,147) while getAuthorizations() (AnyQuery.java:175) and
  getCellVisibility() (AnyMutation.java:292) LEAK HBase's checked DeserializationException |
  AnyOperation.java:128,147 ; AnyQuery.java:175 ; AnyMutation.java:292 | Medium |
  Pick one policy. Recommendation: wrap DeserializationException in an unchecked exception for
  consistency with toJson (the wrapper's stated goal is a "simpler API"), or, if the checked leak is
  intentional, document why these two differ. Verified: toJson try/catch->UncheckedIOException present;
  the two getters declare `throws DeserializationException` with bare delegation.
- getFamilyCellMap() returns the LIVE mutable backing map | AnyMutation.java:191 | Low |
  Sibling getAttributesMap() (AnyOperationWithAttributes.java:115) returns an unmodifiable view;
  mutable/immutable inconsistency. Mirrors HBase, so likely KEEP — but document the live-mutability, or
  return an unmodifiable view for safety. Confirming check: ensure no internal caller relies on mutating
  the returned map.

4. MISPLACED METHODS
None.

5. MISSING METHODS
None. (getACL/setACL split across Query/Mutation is correct per HBase; do not hoist.)

6. DEPRECATION / REMOVAL CANDIDATES
- compareTo(Row) | AnyMutation.java:697 (mirrored on AnyGet.java:992) | High (already @Deprecated) |
  KEEP deprecated; replacement Row.COMPARATOR is documented and consistent. Unavoidable while the
  subclasses `implement Row`.
- Dead commented-out blocks (AnyOperation.java:155-176, AnyMutation.java:195-220) | Low | Delete for
  cleanliness; no API impact.

CROSS-CLASS NOTES
- The wrapper applies an inconsistent checked-exception policy (wrap IOException, leak
  DeserializationException) — this same wrap-vs-leak question recurs across the executors; track it as a
  library-wide convention to settle.
- Self-typed base methods make subclass re-overrides unnecessary; AnyScan violates this for
  setColumnFamilyTimeRange (see AnyScan report) — bases are the correct model.
