1. OVERALL ASSESSMENT
AnyIncrement is a faithful fluent wrapper over HBase `Increment`. No correctness defects. Findings are a
redundant attribute overload, a surprising-but-mirrored equality contract, and cross-class factory parity.
[RE-REVIEW 2026-06-24: Re-verified the full file (797 lines) against live source. All findings below
re-confirmed at the cited (now-current) line numbers. The setAttribute(String,byte[]) overload is now
documented as a "raw-bytes fast path" (AnyIncrement.java:669-704) — this addresses the redundancy
finding to PARTIALLY-ADDRESSED (documented & intentional, still surface-redundant). No new correctness
defects found.]

2. NAMING ISSUES

  AnyPut/AnyDelete rowOffset/rowLength. Converge across the four subclasses (see SUMMARY).
  [RE-REVIEW 2026-06-24: STILL-VALID — of(byte[],int,int) at AnyIncrement.java:225 and its ctor at

  Confirming check: confirm HBase Increment.equals is row-key-only in the resolved hbase-client jar.

6. DEPRECATION / REMOVAL CANDIDATES
- setAttribute(String,byte[]) | AnyIncrement.java:700 | Medium | REDUNDANT — observationally equivalent
  to the inherited base setAttribute(String,Object) (AnyOperationWithAttributes.java:143), because
  toValueBytes returns (byte[])value unchanged for a byte[] (HBaseExecutor.java:3029-3030). Either remove
  it (callers use the inherited Object overload, same result) or — if a documented raw-bytes fast path is
  desired — add it symmetrically to all four subclasses. Recommend remove, for a smaller, uniform surface.
  [RE-REVIEW 2026-06-24: PARTIALLY-ADDRESSED — overload still present at AnyIncrement.java:700, still
  observationally equivalent to base setAttribute(String,Object) (verified: toValueBytes byte[] identity
  at HBaseExecutor.java:3029-3030). HOWEVER the maintainer chose the "documented raw-bytes fast path"
  branch: the Javadoc (AnyIncrement.java:669-698) now explicitly states it "bypasses toValueBytes ...
  makes the byte-array intent explicit at the call site" and cross-references
  AnyOperationWithAttributes#setAttribute(String,Object). It exists on AnyIncrement+AnyAppend but NOT on
  AnyPut/AnyDelete, so the cross-class asymmetry the finding called out REMAINS: to fully resolve, either
  remove it from both or add it to all four. STILL-VALID as a uniformity item; no longer an
  undocumented-redundancy item.]
