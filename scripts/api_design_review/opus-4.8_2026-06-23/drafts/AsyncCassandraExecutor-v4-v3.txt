CROSS-CLASS NOTES
================================================================================
A) v3-vs-v4 DIVERGENCE — all INTENTIONAL except Javadoc drift:
   - execute(...) wrapping (ResultSets.wrap in v4 only) = driver-version requirement,
     not drift. Verified against ResultSets.wrap (v4) and driver-3 ListenableFuture<ResultSet>.
   - Generic ST param: Statement<?> (v4) vs raw Statement (v3); BT: BatchType vs
     BatchStatement.Type — both follow the respective driver type systems. Correct.
   - REAL DRIFT (only finding pair): v3's two NEW BiFunction `stream` overloads are
     MISSING the `@throws IllegalArgumentException if rowMapper is null` Javadoc tag
     that v4 carries (v3:152,193 vs v4:148,189). Bodies are identical
     (N.checkArgNotNull). Low/Medium — doc-only; fix v3 to match v4.

B) ASYNC-vs-SYNC FACTORY NAMING SYMMETRY:
   - There is NO `createAsyncCassandraExecutor` / `createCassandraExecutor` static
     factory anywhere (verified zero matches in all four classes). The async facade
     is reached exclusively via CassandraExecutor.async() (sync side) and the
     package-private ctor (async side). So the "createAsyncX vs createX" symmetry
     question is MOOT — no asymmetry exists because no such factories exist. No action.
   - Accessor symmetry is clean: sync concrete exposes async() (CassandraExecutor.java:475
     / v3:504); async concrete exposes sync() (AsyncCassandraExecutor.java:77 / v3:81),
     each covariantly typed to its own driver-version partner. Good.

C) MISSING-OVERLOAD SYMMETRY (shared, single fix):
   - Add a named-parameter BiFunction streamer (stream(String, BiFunction, Map<String,Object>))
     to BOTH concretes, or document the positional-only limitation. Medium confidence;
     currently no Map-param BiFunction stream exists on either concrete or the base.
