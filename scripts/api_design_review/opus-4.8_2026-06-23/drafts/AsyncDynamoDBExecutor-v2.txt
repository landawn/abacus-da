API DESIGN REVIEW — AsyncDynamoDBExecutor (AWS SDK v2)
File: abacus-da-all/src/main/java/com/landawn/abacus/da/aws/dynamodb/v2/AsyncDynamoDBExecutor.java (public final class implements AutoCloseable, ~3490 lines)
Sync sibling (v2): abacus-da-all/src/main/java/com/landawn/abacus/da/aws/dynamodb/v2/DynamoDBExecutor.java
v1 async cousin:   abacus-da-all/src/main/java/com/landawn/abacus/da/aws/dynamodb/AsyncDynamoDBExecutor.java
Reviewer: opus-4.8   Date: 2026-06-23
Scope: all PUBLIC methods (incl. nested Mapper<T>). REPORT-ONLY (no source modified).

================================================================================
1. OVERALL ASSESSMENT
================================================================================
AsyncDynamoDBExecutor (v2) is a disciplined async facade over the AWS SDK v2
DynamoDbAsyncClient. Unlike the v1 async cousin (which dispatches the synchronous
v1 client through a library AsyncExecutor and returns ContinuableFuture), this
class returns the SDK's native java.util.concurrent.CompletableFuture and is a
deliberate, fully-documented departure from the library's ContinuableFuture
convention — appropriate, because it wraps an inherently non-blocking client
rather than fabricating async over a blocking one.

Operationally it mirrors the v2 sync DynamoDBExecutor's data-plane surface
almost 1:1: getItem / batchGetItem / putItem / batchWriteItem / updateItem /
deleteItem / list / query / stream / scan, plus mapper() and the nested Mapper<T>
with the same overload families, parameter order, and generic shapes. The two
package-private putItem(String,Object[,String]) overloads match the sync class
exactly (and are correctly kept non-public for the documented overload-ambiguity
reason). close() and dynamoDBAsyncClient() are correct.

There are NO must-fix defects. The actionable items are: (a) one genuine
correctness/design concern that the list/query/stream/scan auto-pagination
machinery performs BLOCKING dynamoDBClient.query(...).get()/scan(...).get()
calls inside CompletableFuture continuations and lazy iterators — defensible and
documented, but the single most substantive thing to confirm on an "async" class;
(b) a small return-type asymmetry in Mapper.query (v2 drops the generic <T> that
v1 async kept); (c) the whole-class CompletableFuture-vs-ContinuableFuture
question recorded for the record. Naming, placement, and overload symmetry are
clean.

================================================================================
2. NAMING ISSUES
================================================================================
None. Every method name matches its v2 DynamoDBExecutor counterpart exactly
(getItem, batchGetItem, putItem, batchWriteItem, updateItem, deleteItem, list,
query, stream, scan, mapper, close, dynamoDBAsyncClient). The accessor is named
dynamoDBAsyncClient() (vs sync dynamoDBClient()) — the rename is correct and
intentional because the returned type differs (DynamoDbAsyncClient vs
DynamoDbClient). Mapper method names (getItem/batchGetItem/putItem/batchPutItem/
updateItem/deleteItem/batchDeleteItem/batchWriteItem/list/query/stream/scan)
match the sync Mapper exactly.

3. RETURN-TYPE AND PARAMETER ISSUES
================================================================================

[R1] list / query / stream / scan auto-pagination blocks the calling thread via
     synchronous .get() on the async client
     list(QueryRequest, Class)            | AsyncDynamoDBExecutor.java:1677 (loop body :1691)
     query(QueryRequest, Class)           | AsyncDynamoDBExecutor.java:1795 (loop body :1812)
     stream(QueryRequest, Class)          | AsyncDynamoDBExecutor.java:1934 (iterator :1951)
     scan(ScanRequest, Class)             | AsyncDynamoDBExecutor.java:2283 (iterator :2300)
     | Confidence: Medium (design concern, not a bug — it works and is documented)
     | Observation: list/query call thenApplyAsync(...) and then, for subsequent
       pages, loop calling dynamoDBClient.query(newQueryRequest).get() — a BLOCKING
       wait on the native async client from within a ForkJoinPool/common-pool stage.
       stream/scan build an ObjIterator whose hasNext() does dynamoDBClient.query(
       ...).get()/scan(...).get() synchronously on the consuming thread. So the
       headline "non-blocking" promise holds only for the FIRST page; multi-page
       results block a pool thread (list/query) or the consumer thread (stream/scan).
       The Javadoc is honest about this for stream/scan ("each page is fetched
       synchronously inside the consuming thread … terminal operations block",
       :1922-1926, :2272-2275) but list()/query() Javadoc does NOT carry the same
       blocking warning for their page-2..N .get() calls.
     | Recommendation: keep behavior, but (1) add the same "subsequent pages are
       fetched with a blocking get()" note to list(QueryRequest,Class) and
       query(QueryRequest,Class) Javadoc for parity with stream/scan; and (2)
       consider a truly-async pagination via thenCompose() chaining over
       dynamoDBClient.query(...) instead of .get() so no pool thread is parked —
       this is the one place the v2 async class falls short of "fully non-blocking."
     | Confirming check (Medium->Low): this exactly mirrors the sync class's loop
       (DynamoDBExecutor.java:2973-2990 list, 3093 query) so the .get() is a faithful
       port; the design question is whether an async-branded API should ever block
       on .get(). Confirm intended scope before changing — the mirror is by-design,
       the missing list/query Javadoc note is the concrete gap.

[R2] Mapper.query(QueryRequest) returns CompletableFuture<Dataset> but loses the
     <T> generic that the v1 async Mapper.query exposes
     Mapper.query(QueryRequest)           | AsyncDynamoDBExecutor.java:3195
     executor.query(QueryRequest, Class<?>)| AsyncDynamoDBExecutor.java:1795
     | Confidence: Low (intra-v2 consistent; only an across-generation divergence)
     | Observation: the executor-level query(QueryRequest, Class<?>) takes Class<?>
       (raw, non-generic) and Mapper.query passes targetEntityClass into it. This is
       INTERNALLY consistent with the v2 sync class (DynamoDBExecutor.java:3093
       query(QueryRequest, Class<?>) and Mapper.query :4162 also Class<?>). The v1
       async cousin uses a generic <T> query(QueryRequest, Class<T>) (v1
       AsyncDynamoDBExecutor.java:1674). So this is a v1-vs-v2 divergence, not a
       sync-vs-async mismatch.
     | Recommendation: no change. Dataset is not parameterized by T, so Class<?> is
       the honest signature; v2 deliberately dropped the misleading <T>. Recorded
       only so the v1/v2 difference is not mistaken for an async regression.
     | Confirming check: none needed — verified both v2 classes use Class<?> and
       both v1 classes use Class<T>.

Verified CONSISTENT (explicitly NOT findings):
 - getItem typed overloads return CompletableFuture<T> resolving to null on miss
   (:570,:617,:678) — faithful mirror of sync T getItem (:2119,:2154,:2182). No
   Optional-inside-future anywhere; neither class uses Optional. Correct.
 - putItem/updateItem/deleteItem return CompletableFuture<...Response> mirroring
   the sync ...Response return (PutItemResponse/UpdateItemResponse/
   DeleteItemResponse/BatchWriteItemResponse). v2-native response types, not v1's
   *Result. Correct for the SDK generation.
 - Parameter ordering matches sync v2 on every overload family (tableName, key/
   item, consistentRead/returnValues, targetClass last). No reordering.
 - No per-call Executor parameter on any method. Correct: the async client owns its
   own executor (asyncConfiguration), so a library Executor param would be
   meaningless here — and the class Javadoc says so (:106-112). Contrast: v1 async
   is backed by a constructor AsyncExecutor; v2 has none by design.
 - Mapper.checkItem(BatchGetItemRequest)/checkItem(BatchWriteItemRequest) only
   VALIDATE the table name and do not substitute it (:3413,:3424), whereas the
   single-table request checks substitute an empty table (:3403,:3434,:3444,:3454).
   This matches the sync Mapper exactly and is correct (batch requests are
   multi-table maps with no single slot to fill). Not a finding.

================================================================================
CROSS-CLASS NOTES (symmetry with sync v2; async return-type convention; v1 vs v2)
================================================================================
- ASYNC RETURN TYPE — DELIBERATE CONVENTION BREAK. Every public method returns
  java.util.concurrent.CompletableFuture, NOT the library's ContinuableFuture.
  This is the central design fact and it is intentional and well-documented
  (class Javadoc :78-167, :119-127). It diverges from the v1 async cousin, which
  returns ContinuableFuture and is backed by a library AsyncExecutor (v1
  AsyncDynamoDBExecutor.java:37-100). Rationale is sound: v2 wraps a natively
  non-blocking client, so surfacing its own CompletableFuture avoids a pointless
  bridging layer. NOT flagged as a defect — recorded so the inconsistency with the
  library-wide ContinuableFuture convention is a conscious, visible choice. (If the
  library ever wants uniformity, a ContinuableFuture.wrap(...) adapter on each
  return would reconcile it without changing semantics.)
- DATA-PLANE SYMMETRY with v2 sync is essentially 1:1: identical method names,
  overload families, parameter order, and generics; sync `T` -> CompletableFuture<T>,
  sync `Stream<X>` -> CompletableFuture<Stream<X>>, sync `Dataset` ->
  CompletableFuture<Dataset>, sync `...Response` -> CompletableFuture<...Response>.
  The two package-private putItem(String,Object[,String]) overloads match the sync
  class and stay non-public for the same documented overload-ambiguity reason
  (:1158,:1184).
- THE ONE ASYNC-PURITY GAP (R1): list/query block a pool thread and stream/scan
  block the consumer thread when fetching pages 2..N via .get() on the async
  client. Faithful port of the sync loop, documented for stream/scan, under-
  documented for list/query. Only place the class isn't truly non-blocking.
- v1-vs-v2 DIVERGENCES (all acceptable): (1) CompletableFuture vs ContinuableFuture;
  (2) native v2 *Response types vs v1 *Result; (3) no constructor AsyncExecutor /
  no per-call Executor param in v2 (client owns its executor) vs v1's AsyncExecutor
  backbone; (4) Mapper.query Class<?> (v2) vs Class<T> (v1) — R2.
- No @Beta in either v2 class; the single sync @Deprecated (ConditionBuilder.create)
  is in a static helper not mirrored here. Deprecation/@Beta parity is clean.

OVERALL: healthy, consistent async facade. No must-fix items. R1 (blocking
.get() during multi-page list/query/stream/scan + the missing list/query Javadoc
note) is the only substantive thing to confirm/act on; the CompletableFuture-vs-
ContinuableFuture choice is deliberate; R2 and M1/M2 are informational.
