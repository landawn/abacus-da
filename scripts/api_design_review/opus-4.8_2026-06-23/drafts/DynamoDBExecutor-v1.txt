API DESIGN REVIEW — DynamoDBExecutor (AWS SDK v1)
File: abacus-da-all/src/main/java/com/landawn/abacus/da/aws/dynamodb/DynamoDBExecutor.java (~5064 lines)
Twins: aws/dynamodb/v2/DynamoDBExecutor.java (SDK v2), aws/dynamodb/AsyncDynamoDBExecutor.java (async)
Reviewer model: opus-4.8   Date: 2026-06-23

================================================================================
RELATIONSHIP NOTE (read first)
================================================================================
This is a thin, by-design wrapper over the AWS SDK v1 AmazonDynamoDB API. Methods
named getItem/putItem/updateItem/deleteItem/batchGetItem/batchWriteItem mirror the
SDK and return the SDK's *Result types (PutItemResult, etc.) — this is intentional
and matches the established "SDK-mirror" convention used by HBaseExecutor (which
returns raw Result, not Optional). It does NOT follow the get=Optional / gett=null
discipline used by MongoCollectionExecutor, and that is acceptable here BECAUSE the
class is an SDK-mirror, not a domain finder. The findings below flag only genuine
inconsistencies (internal drift, v1/v2 drift, naming inaccuracies), not SDK mirroring.

================================================================================
1. OVERALL
================================================================================
Solid, internally consistent SDK-v1 wrapper. The overload families are systematic
(raw-Map / typed-Class / full-Request triplets for every CRUD op), Javadoc is
thorough and honest about sharp edges (unprocessed-keys silently dropped,
auto-pagination ignoring Limit, ConditionBuilder.build() draining itself). Most
real findings are (a) v1-vs-v2 naming/overload drift and (b) a small number of
return-type / generics inconsistencies. No correctness bugs in scope; this is a
design-consistency pass only.

================================================================================
2. NAMING
================================================================================

[2a] toAttributeValue / toAttributeValueUpdate  (static converters)
    DynamoDBExecutor.java:906, 949, 982  | HIGH
    The v2 twin renamed the identical-semantics converters to attrValueOf /
    attrValueUpdateOf (v2:442,498,556). Same library, same author, same concept,
    two different names across the twins. This is accidental API drift. The
    `as*`/`to*` split here is also slightly off: `asKey/asItem/asUpdateItem` build
    maps while `toAttributeValue/toItem/toMap/toEntity` convert single values —
    naming is internally consistent in v1, the issue is ONLY the cross-twin drift.
    Rec: pick one name pair for both twins (keep `toAttributeValue`/
    `toAttributeValueUpdate` as the more descriptive, or align v2's `attrValueOf`).
    Not back-compat-free, but worth recording as the canonical drift item.

[2b] putItem(String,Object[]) family naming — N/A (package-private)
    The Object-entity putItem overloads (2296, 2300) are package-private with an
    explicit comment explaining why no public entity overload exists ("Object is
    ambiguous"). Reasonable; no action.

No other naming findings — query/scan/list/stream/get/put/update/delete/batch are
all used accurately and match the verb semantics.

================================================================================
3. RETURN TYPE / PARAMETERS
================================================================================

[3a] query(QueryRequest, Class<?>) uses raw wildcard, not Class<T>
    DynamoDBExecutor.java:2815 (and Mapper.query 4043)  | MEDIUM
    Every other typed entry point takes Class<T> and returns a parameterized
    result; query(...) takes Class<?> and returns a non-generic Dataset. That is
    actually correct FOR Dataset (Dataset is untyped), but the parameter being
    `Class<?>` rather than `Class<T>` is inconsistent with toList/list/stream/scan
    which all use `Class<T>`. Since the class only drives row-mapping and the
    return is Dataset, `Class<?>` is defensible — flagging as LOW-confidence
    consistency nit, not a bug. Confirming check: intentional? compare to
    query(QueryRequest) which delegates with Clazz.PROPS_MAP.

[3b] getItem(...) returns bare nullable T / Map, no Optional/gett twin
    DynamoDBExecutor.java:1865,1903,1925,1964,1990,2013 (+ Mapper 3477..3552) | LOW
    Per the relationship note this is the SDK-mirror convention (HBase does the
    same). NOT recommending an Optional overload — recording only so the next
    reviewer doesn't re-flag it as a get=Optional violation. No action.

[3c] batchGetItem(Map,...) silently drops UnprocessedKeys
    DynamoDBExecutor.java:2059,2080,2130,2152 | MEDIUM (documented, but still a trap)
    The Map-returning batchGetItem overloads return only getResponses() and discard
    getUnprocessedKeys(). Javadoc is explicit (2031-2036), and the *Request overload
    is offered as the escape hatch. Ground rule 5 says don't excuse an inconsistent
    issue just because it's documented — but here the behavior is a deliberate
    contract difference between the convenience overload and the Request overload,
    not an inconsistency. Recording as a known sharp-edge; no change recommended
    beyond what is already documented.

[3d] Mapper.batchPutItem / batchDeleteItem do not split >25-item collections
    DynamoDBExecutor.java:3715, 3936 | LOW
    Documented (throws AmazonServiceException past 25). Mapper.batchGetItem(>100)
    likewise un-split (3561-3563). Consistent across the three; by-design. No action.

================================================================================
4. MISPLACED / CLASS PLACEMENT
================================================================================

[4a] Static pure converters live on the executor instance class
    toItem/toUpdateItem/toMap/toEntity/toAttributeValue/toAttributeValueUpdate/
    asKey/asItem/asUpdateItem/toList/extractData — DynamoDBExecutor.java:526..1818 | LOW
    These are all static, stateless, AmazonDynamoDBClient-free helpers. They could
    live on a `DynamoDB`/`DynamoDBUtil` util class, leaving the executor focused on
    I/O. BUT: the v2 twin co-locates the identical set on its executor, and callers
    reach them as DynamoDBExecutor.asKey(...) throughout the Javadoc examples.
    Moving them is a large back-compat break for a cosmetic gain. Record as a
    placement observation; do NOT recommend moving (consistency with v2 + heavy
    documented usage outweigh purity).

[4b] Filters / ConditionBuilder nested in DynamoDBExecutor
    DynamoDBExecutor.java:4381 (Filters), 4718 (ConditionBuilder) | LOW
    Both are self-contained static helper classes with no executor dependency.
    Same placement is mirrored in v2 (v2:Filters, ConditionBuilder). Acceptable as
    a discoverability choice (Filters.eq read from the executor namespace). No action.

================================================================================
5. MISSING METHODS
================================================================================

[5a] No toEntity(GetItemResult, Class<T>) overload  — v1/v2 GAP
    DynamoDBExecutor.java (absent)  vs  v2:1263 toEntity(GetItemResponse, Class) | HIGH
    The v2 twin provides toEntity(GetItemResponse,...) that null/has-item-guards and
    unwraps the response, in addition to toEntity(Map,...). v1 has ONLY
    toEntity(Map<String,AttributeValue>, Class) (1319). A caller holding a raw
    GetItemResult from dynamoDBClient() must manually call .getItem() first and
    null-check. Adding `static <T> T toEntity(GetItemResult, Class<T>)` would close
    the drift and is purely additive (no break).
    Confirming check: grep v1 for `GetItemResult` param in any public static — none.

[5b] No async() on v2 twin — informational, not a v1 gap
    v1 has async() (505) returning AsyncDynamoDBExecutor; v2 has no async() and no
    AsyncDynamoDBExecutor for SDK v2 (SDK v2 has native async clients). This is an
    intentional SDK-version difference, recorded under cross-class notes — NOT a
    missing method on v1.

[5c] No offset/count list/query overloads (commented out)
    DynamoDBExecutor.java:2688-2735, 2837-2877 | LOW
    list(QueryRequest, pageOffset, pageCount, Class) and the paged scan/query
    variants are commented out (page-offset semantics over LastEvaluatedKey are
    leaky). The static toList(QueryResult, offset, count, Class) (1592) and
    extractData(.., offset, count) (1722/1768) DO exist for single-page slicing.
    Deliberately omitted; the streaming scan + auto-paginating list cover the need.
    No action — record only.

6. DEPRECATION
================================================================================

[6a] ConditionBuilder.create()  — already @Deprecated
    DynamoDBExecutor.java:4736 | (informational)
    Correctly deprecated in favor of Filters.builder() (4694). Note: the SAME
    deprecated create() also exists in the v2 twin (v2:4831) — keep them in lockstep
    when eventually removed. No action beyond confirming the replacement pointer is
    present (it is).

No other deprecation candidates. The package-private Object-entity putItem overloads
(2296/2300) are not public, so out of scope for deprecation.

================================================================================
CROSS-CLASS NOTES (v1 vs v2 divergence; library convention deviations)
================================================================================
1. CONVERTER NAMING DRIFT (HIGH): v1 toAttributeValue/toAttributeValueUpdate
   vs v2 attrValueOf/attrValueUpdateOf — identical semantics, divergent names.
   Pick one canonical pair for both twins. [§2a]

2. MISSING toEntity(Response/Result) OVERLOAD (HIGH): v2 has
   toEntity(GetItemResponse, Class); v1 lacks toEntity(GetItemResult, Class).
   Additive fix. [§5a]

3. RETURN-TYPE FAMILIES MIRROR THE SDK, NOT THE Optional/gett DISCIPLINE: getItem
   returns nullable T (like HBaseExecutor), NOT Optional (unlike MongoCollection*).
   This is correct for an SDK-mirror executor; do not "fix". [§3b]

4. ASYNC: v1 exposes async() -> AsyncDynamoDBExecutor returning ContinuableFuture
   (matches the library's ContinuableFuture async convention — verified in
   AsyncDynamoDBExecutor: ContinuableFuture<...> on every op). v2 has no async()
   (SDK v2 supplies native async clients). Intentional SDK-version difference. [§5b]

5. PARAMETER GENERICS: query(QueryRequest, Class<?>) uses a wildcard where the rest
   of the typed surface uses Class<T>; defensible for an untyped Dataset return but
   inconsistent with the toList/list/stream/scan(Class<T>) family. LOW. [§3a]

6. STATIC HELPER PLACEMENT and Filters/ConditionBuilder nesting are identical
   between v1 and v2 — consistent drift-free, just not "pure". No action. [§4]
