API DESIGN REVIEW — MongoCollectionMapper<T>
File: abacus-da-all/src/main/java/com/landawn/abacus/da/mongodb/MongoCollectionMapper.java (~3355 lines)
Reviewer: opus-4.8  Date: 2026-06-23
Relationship: typed entity facade over MongoCollectionExecutor (raw Document). Reactive twin at
  mongodb/reactivestreams/MongoCollectionMapper.java. Scope = all PUBLIC methods (incl. deprecated).

CONVENTION BASELINE (verified against source, not a finding):
- get/findFirst -> Optional<T>; gett -> nullable T; findOneAndXxx -> nullable T; list -> List<T>;
  query -> Dataset; stream/distinct/aggregate/groupBy* -> Stream<T>; writes -> driver result objects.
- Projection/filter/sort/offset/count ordering is uniform across list/query/stream/findFirst
  (selectPropNames FIRST, then filter, then sort, then offset/count). get()/gett() are id-first then
  projection. Both orderings are internally consistent — NOT a finding.
- Almost every method is a thin 1:1 delegation to the executor with rowType bound; distinct(...) is the
  one exception (re-implemented via a $group/$project pipeline — see §1).

=====================================================================================================
1. FUNCTIONALITY
=====================================================================================================

[F1] distinct(String) / distinct(String,Bson) re-implement a pipeline; groupBy*/aggregate delegate raw
     | MongoCollectionMapper.java:3112-3133 (distinct), 3161-3166; vs groupBy 3225-3256, groupByAndCount 3286-3320
     | Confidence: Low (correctness note, not an API-surface defect)
     | distinct() deliberately routes through distinctPipeline() because the executor's native
       distinct(field, entityClass) decodes each scalar VALUE with the entity codec and throws
       BsonInvalidOperationException (per the inline comment at 3118-3121). groupBy/groupByAndCount,
       which return the same "entity carrying one/few fields" shape, instead delegate straight to
       collectionExecutor.groupBy(fieldName, rowType). They are documented as safe because the executor
       wraps each group as a {field: value} document — but the asymmetry (one family hand-rolls the
       pipeline, the sibling families trust the executor) is worth a confirming check.
     | Recommendation: confirm executor.groupBy/groupByAndCount/aggregate already emit decodable
       {field:value} documents for scalar group keys (i.e. don't hit the same BsonInvalidOperationException
       distinct was patched to avoid). If they do, no change. If not, route them through a pipeline too,
       OR (cleaner) push the distinct() pipeline fix DOWN into the executor so the mapper can delegate
       uniformly and the reactive twin inherits the same fix.
     | Low -> confirming check: run distinct("category") and groupBy("category") against a string field on
       real Mongo; both should return populated entities, not throw.

[F2] query(...) family is entity-bound but returns an untyped Dataset
     | MongoCollectionMapper.java:1474-1677 (9 overloads)
     | Confidence: Low
     | rowType is passed to executor.query(...) but the return is a Dataset (tabular, column-name keyed),
       so the entity type only influences column extraction/ordering, never the row type. This mirrors the
       executor and is internally consistent; flagged only because a reader of a "typed mapper" may expect
       a typed result. No change recommended — documented and symmetric. (Confirming: Dataset has no T.)

3. RETURN TYPE
=====================================================================================================

[R1] findOneAndUpdate / findOneAndReplace / findOneAndDelete return nullable T, while get/findFirst
     return Optional<T>
     | MongoCollectionMapper.java:2865, 2898, 2930, 2961, 2989, 3021, 3053, 3083
     | Confidence: Low
     | Single-entity reads are split: get/findFirst -> Optional<T> (empty == not found), but the

       with gett. Keep as-is; documenting here only so the Optional-vs-null split is a conscious decision,
       not an oversight.
     | Low -> confirming check: none needed; behavior matches Javadoc.

5. MISSING METHODS / CLASS PLACEMENT
=====================================================================================================

[M1] No whole-collection convenience overloads: list(), stream(), query() (no-arg, all entities)
     | MongoCollectionMapper.java — absent (cf. executor stream() @ MongoCollectionExecutor.java:2041)
     | Confidence: Medium
     | The executor exposes a no-arg stream() that returns every document; the mapper exposes no no-arg
       list()/stream()/query(). To fetch/stream the entire collection a caller must pass a sentinel filter
       (e.g. new Document() or Filters.empty()). Every list/stream/query overload requires a filter.
     | Recommendation: add list()/stream()/query() (and optionally stream(int offset,int count)) that
       delegate with an empty filter, OR add a stream(Class)-style all-docs path mirroring the executor's
       stream(). Mirror the same addition into the reactive twin to keep them symmetric.
     | NOTE: the reactive twin ALSO lacks these no-arg overloads, so the two facades are currently
       symmetric in their omission — fix both together or neither.

[M2] findFirst has no (Bson filter, Bson sort) overload — sort without projection requires a null arg
     | MongoCollectionMapper.java:629-731 (findFirst family)
     | Confidence: Low
     | To get a deterministic "first" with a sort but no projection, callers must call
       findFirst((Collection<String>) null, filter, sort) or findFirst((Bson) null, filter, sort). A
       findFirst(Bson filter, Bson sort) shortcut is missing. This exactly mirrors the executor and the
       reactive twin (both also require the null-projection form), so it is a deliberate, symmetric gap.
     | Recommendation: optional convenience overload; low priority precisely because it is symmetric with
       executor + reactive twin. If added, add to all three.

[M3] bulkWrite(...) accepts driver WriteModel<? extends Document> — breaks the typed abstraction
     | MongoCollectionMapper.java:2806, 2835 (vs typed bulkInsert @ 2747, 2776)
     | Confidence: Low (class-placement / by-design tension)
     | In a typed mapper, bulkWrite forces the caller to hand-build Document-based InsertOneModel/
       UpdateOneModel/DeleteOneModel etc. — there is no entity-typed WriteModel path, unlike bulkInsert
       which is entity-typed. This is the same untyped escape hatch the executor offers and is arguably
       fine as a raw passthrough (the Javadoc is explicit), but it is the one write method that does not
       project T. Reasonable to leave on the mapper as a passthrough; flagged only so the typed-vs-raw
       split is intentional.
     | Low -> confirming check: none; matches executor signature by design.

CROSS-CLASS NOTES
=====================================================================================================

SYMMETRY vs reactive twin (mongodb/reactivestreams/MongoCollectionMapper.java):
- The reactive twin collapses get(...) AND gett(...) into a single Mono<T> family (Mono is the async
  Optional) and therefore has NO gett() overloads and NO Optional<T> returns. This is the correct,
  deliberate asymmetry — do NOT try to add gett() to the reactive twin or Mono-ize the sync gett.
- findFirst: sync returns Optional<T> (4 overloads), reactive returns Mono<T> (4 overloads) — symmetric.
- The reactive twin lacks the primitive OptionalXxx accessors' "present-with-primitive-default" nuance:
  it returns Mono<Boolean>/Mono<Integer>/... i.e. plain boxed wrappers. So the sync mapper's
  queryForBoolean -> OptionalBoolean (present-but-default-on-missing-field) semantics have NO reactive
  analogue for the empty-vs-present-default distinction. Acceptable (Mono.empty == no match), but note
  the sync-only "JDBC primitive default surfaces in a present Optional" quirk does not translate.
- distinct/aggregate/groupBy/groupByAndCount: sync Stream<T> vs reactive Flux<T> — symmetric. The
  distinct() pipeline re-implementation (§F1) exists ONLY in the sync mapper here; verify the reactive
  twin's distinct() applies the SAME scalar-decoding workaround (it delegates differently) — if the fix
  lives only in sync, the reactive twin may still throw BsonInvalidOperationException on scalar fields.
  >>> This is the single most important cross-class check: confirm reactive distinct() handles scalars.
- No-arg list()/stream()/query() (§M1) and findFirst(filter,sort) (§M2): both facades omit them — fix in
  both or neither.

MAPPER vs EXECUTOR surface:
- The mapper is a faithful typed projection of the executor: it binds rowType and drops the executor's
  Document-returning and explicit-Class<T> overloads (get(objectId) -> Document, get(objectId, Class),
  list(Bson) -> List<Document>, stream() / stream(Class), etc.). The only behavioral divergence is
  distinct() (§F1), which the mapper re-implements instead of delegating.
- The executor's no-arg stream() (all docs) is NOT projected onto the mapper (§M1) — the main concrete
  gap in the "consistent typed projection" claim.
- query(...) returns Dataset on BOTH classes; the mapper's rowType only drives column extraction (§R2/R1).

RETURN-CONVENTION SUMMARY (consistent and intentional):
  exists->boolean | count->long | get/findFirst->Optional<T> | gett/findOneAndXxx->T-or-null |
  list->List<T> | query->Dataset | stream/distinct/aggregate/groupBy*->Stream<T> |
  insertOne/insertMany->void | bulkInsert->int | updateXxx/replaceXxx->UpdateResult |
  deleteXxx->DeleteResult | bulkWrite->BulkWriteResult.

OVERALL: Surface is clean, well-documented, and a faithful typed projection of the executor. No
high-confidence defects. The actionable items are M1 (missing all-collection overloads, Medium) and the
F1/cross-class distinct-vs-groupBy scalar-decoding consistency check (Low, but with a real correctness
angle for the reactive twin). Everything else is intentional convention to preserve.
