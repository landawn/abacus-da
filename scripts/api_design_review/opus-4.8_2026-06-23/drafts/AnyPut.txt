================================================================================
1. OVERALL ASSESSMENT
================================================================================
AnyPut is a well-built, carefully-documented fluent wrapper over org.apache.hadoop.hbase.client.Put. The factory (`of`) family, the entity-mapping `create` family, the `addColumn`/`add` write family, `toPut` batch converters, `val()`, and identity-based hashCode/equals/toString are all coherent and mirror the native Put plus the sibling Any* wrappers. The few actionable findings are about cross-class CONSISTENCY rather than correctness: AnyPut lacks the `of(byte[])` / `of(byte[], int, int)` byte-array row-key factories that AnyAppend and AnyIncrement both expose, and it lacks the `setAttribute(String, byte[])` convenience overload that both siblings declare; conversely AnyPut carries several `of` overloads (rowIsimmutable, ByteBuffer, partial-row-with-timestamp) that the siblings do not, so the factory surface across the four mutations is uneven. There is also a genuine swapped-argument hazard in the `addColumn(byte[], byte[], ...)` overloads (two adjacent same-typed byte[] family/qualifier params) shared with the siblings, and one minor naming/return inconsistency where the partial-row factory parameters are named `rowOffset`/`rowLength` here but `offset`/`length` in the siblings. None of these is a defect in isolation; they are consistency/completeness items for the maintainer to weigh.

================================================================================
2. NAMING ISSUES
================================================================================

[N1] of(Object, int, int) / of(Object, int, int, long) — partial-row param names
     | AnyPut.java:370, AnyPut.java:411 | Confidence High
     Recommendation: parameters are named `rowOffset` / `rowLength` here, but the
     equivalent slice factories on the siblings name them `offset` / `length`
     (AnyAppend.java:269 `of(byte[], int, int)`, AnyIncrement.java:225). Pick ONE
     convention across the four Any* mutations. `rowOffset`/`rowLength` is the more
     descriptive pair and matches the underlying Put constructor doc; recommend
     renaming the sibling params to `rowOffset`/`rowLength` rather than the reverse.
     (This is a cross-class naming-consistency item, surfaced from AnyPut.)

No other naming defects. `of`, `create`, `addColumn`, `add`, `val`, `toPut` are
all accurate to their behavior.

3. RETURN-TYPE AND PARAMETER ISSUES
================================================================================

[P1] addColumn(byte[] family, byte[] qualifier, byte[] value)
     addColumn(byte[] family, byte[] qualifier, long ts, byte[] value)
     | AnyPut.java:1158, AnyPut.java:1196 | Confidence Medium
     Two adjacent parameters of the SAME type (byte[] family, byte[] qualifier)
     are trivially swappable at a call site with no compile-time protection; a
     swap silently writes to the wrong family/qualifier. This is inherent to
     mirroring Put and is shared by AnyAppend.addColumn(byte[],byte[],byte[])
     (AnyAppend.java:536) and AnyDelete, so it is NOT AnyPut-specific and removal
     is not warranted. Recommendation: keep, but ensure the Javadoc ordering note
     ("family first, qualifier second") stays prominent. No API change.
     (Logged once for the whole byte[] addColumn family per the grouping rule.)

[P2] addColumn(byte[] family, ByteBuffer qualifier, long ts, ByteBuffer value)
     | AnyPut.java:1234 | Confidence Medium
     Asymmetric/incomplete overload: this is the ONLY ByteBuffer-based addColumn,
     and it exists only in the 4-arg (explicit-ts) form. There is no
     addColumn(byte[], ByteBuffer, ByteBuffer) (no-ts) counterpart, unlike the
     byte[] family which provides both ts and no-ts forms (1158/1196). Also the
     family stays byte[] while qualifier/value switch to ByteBuffer, which is an
     odd mix. Recommendation: either add the missing no-timestamp ByteBuffer
     overload for parity, or (preferred) drop this niche overload entirely since
     callers with ByteBuffers can pass byte[] — confirm usage before removing.
     Confirming check (Low part): grep call sites of the ByteBuffer addColumn
     across the repo/tests; if unused, remove it.

[P3] create(Object, Collection<String>) and create(Object, Collection<String>, NamingPolicy)
     — null selectPropNames silently means "all properties"
     | AnyPut.java:773, AnyPut.java:804 | Confidence Low
     A null `selectPropNames` is documented (line 767/797) to include ALL
     properties, which overlaps the semantics of create(Object) / create(Object,
     NamingPolicy). This is a documented convenience but a mild footgun (a null
     filter computed upstream silently writes every column). Recommendation: keep
     the behavior (it is intentional and consistent with abacus conventions) but
     consider it acceptable as-is; no change required.
     Confirming check: verify create(Object, NamingPolicy) at 654 and the null
     branch at 808 produce identical output — they do (808 delegates to 654).

Return types across the class are correct: factories return AnyPut / List<AnyPut>;
val() returns the live Put (never null, documented); toPut returns List<Put>;
fluent writers return `this`. No Optional-vs-null or array-vs-List concerns.

None. Every public method declared in AnyPut belongs on AnyPut: the Put-specific

5. MISSING METHODS (with target class)
================================================================================

[M1] of(byte[] rowKey)  AND  of(byte[] rowKey, int offset, int length)
     | target class: AnyPut | Confidence High
     AnyPut has NO byte[]-typed row-key factory. Both siblings provide them:
     AnyAppend.of(byte[]) (AnyAppend.java:237), AnyAppend.of(byte[],int,int)
     (AnyAppend.java:269); AnyIncrement.of(byte[]) (AnyIncrement.java:188),
     AnyIncrement.of(byte[],int,int) (AnyIncrement.java:225). A caller holding a
     pre-encoded byte[] row key must currently route through of(Object), which
     re-runs toRowBytes on a byte[] (extra dispatch/branch) and is less explicit.
     Recommendation: add public static AnyPut of(byte[] rowKey) and
     of(byte[] rowKey, int rowOffset, int rowLength) for parity with the siblings
     and with the native Put(byte[]) / Put(byte[],int,int) constructors.
     (Note: AnyPut.of(Object,int,int) at 370 already covers the slice case for
     Object keys, but a typed byte[] overload avoids the Object boxing/dispatch.)

[M2] setAttribute(String name, byte[] value)
     | target class: AnyPut | Confidence Medium
     Both AnyAppend (AnyAppend.java:635) and AnyIncrement (AnyIncrement.java:700)
     declare a setAttribute(String, byte[]) override returning their own concrete
     type and making the byte[] intent explicit at the call site. AnyPut does not,
     so on AnyPut a byte[] argument falls through to the base
     AnyOperationWithAttributes.setAttribute(String, Object) (AnyOperationWith-
     Attributes.java:143). Behavior is equivalent (toValueBytes returns byte[]
     as-is), but the surface is INCONSISTENT across the three writable mutations.
     Recommendation: for symmetry either add setAttribute(String, byte[]) to
     AnyPut, OR remove it from AnyAppend/AnyIncrement and rely on the base Object
     overload everywhere. (The siblings' own Javadoc admits it is "observationally
     equivalent" to the base method — leaning toward removing the sibling
     overloads for a smaller, uniform surface.)

[M3] (consider) of(byte[] rowKey, long timestamp, NavigableMap<byte[],List<Cell>> familyMap)
     | target class: AnyPut | Confidence Low
     AnyAppend (AnyAppend.java:301) and AnyIncrement (AnyIncrement.java:262) both
     offer a (byte[], long, familyMap) bulk-construction factory; AnyPut does not.
     Put has the matching constructor. This is a niche reconstruct-from-cells path.
     Recommendation: add only if the family-map construction path is actually used
     for Puts; otherwise leave out.
     Confirming check: grep for Put(byte[], long, NavigableMap) / familyMap-based
     Put construction usage; add the factory only if there is demand.

CROSS-CLASS NOTES (vs AnyAppend / AnyIncrement / AnyDelete / base chain)
================================================================================

- Row-key factory surface is UNEVEN across the four mutations:
  * AnyPut: of(Object), of(Object,long), of(Object,int,int),
    of(Object,int,int,long), of(Object,boolean), of(Object,long,boolean),
    of(ByteBuffer), of(ByteBuffer,long), of(Put). No byte[] overloads. (M1)
  * AnyAppend/AnyIncrement: of(Object), of(byte[]), of(byte[],int,int),
    of(byte[],long,NavigableMap), of(Append|Increment). No ByteBuffer,
    no rowIsImmutable, no (Object,long)/(Object,long,boolean) timestamp forms.
  * AnyDelete (AnyDelete.java:268-459): of(Object), of(Object,long),
    of(Object,int,int), of(Object,int,int,long), of(Object,long,NavigableMap),
    of(Delete). Object-keyed like AnyPut; also lacks byte[] overloads.
  Net: AnyPut+AnyDelete are "Object-key" oriented; AnyAppend+AnyIncrement are
  "byte[]-key" oriented. Recommend converging on a common minimum set (Object +
  byte[] + slice + copy) so users get the same construction options regardless of
  mutation type. This is the single biggest consistency gap touching AnyPut.

- Slice-param naming differs: AnyPut/AnyDelete use rowOffset/rowLength;
  AnyAppend/AnyIncrement use offset/length. (N1)

- setAttribute(String, byte[]) exists on AnyAppend (635) and AnyIncrement (700)
  but NOT AnyPut. (M2) Standardize in one direction.

- add(Cell): AnyPut.add (1369) and AnyIncrement.add (356) both declare
  `throws IOException`; AnyAppend.add (604) does NOT (Append.add swallows the
  row-mismatch IOException internally, per its Javadoc 582-586). This divergence
  is rooted in the native HBase classes (Put.add/Increment.add throw; Append.add
  swallows), so it is by-design mirroring, NOT an AnyPut defect — noted for
  awareness only.

- addColumn value semantics correctly differ by type and are well documented:
  AnyPut overwrites same (family,qualifier[,ts]); AnyAppend concatenates;
  AnyIncrement adds a long delta. AnyDelete uses addColumn (single version) vs
  addColumns (all versions) — a distinction that does not apply to Put. No
  overlap/duplication after accounting for mirroring.

- equals/hashCode are identity-based on the wrapped native object in all four
  wrappers (AnyPut 1396/1431, AnyAppend 662/694, AnyIncrement 730/764),
  consistent and correctly documented (incl. the Increment row-key-only caveat).
