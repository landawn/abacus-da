RELATIONSHIP MODEL (established before findings)
- All four (AnyPut, AnyDelete, AnyAppend, AnyIncrement) extend
  AnyMutation<AM> <- AnyOperationWithAttributes<AOWA> <- AnyOperation<...>.
- Inherited (NOT flagged here): setTimestamp/getTimestamp, setDurability/get,
  setTTL/get, setACL, setCellVisibility, setClusterIds, get/has families,
  getFamilyCellMap, cellScanner, size/numFamilies/isEmpty, heapSize, getRow,
  compareTo (already @Deprecated in base), getFingerprint, and
  setAttribute(String, Object)/getAttribute/getAttributesMap/setId/setPriority.
- By-design and NOT defects: fluent setters returning `this`/AM; one-to-one
  method mirroring of the wrapped HBase Delete/Increment/Append; val() accessor;
  identity-based equals/hashCode delegating to the wrapped HBase object.
- addColumn vs addColumns in AnyDelete are DISTINCT HBase semantics (single
  version vs all versions); NOT a duplication.


################################################################################
CLASS 1 — AnyDelete
################################################################################

1. OVERALL ASSESSMENT
AnyDelete is a clean, faithful, well-documented mirror of HBase Delete. The
addFamily / addFamilyVersion / addColumn / addColumns families each map to a
distinct, correctly-documented HBase deletion semantic (single vs all versions,
<= vs exact timestamp), and the Javadoc is unusually explicit about the
LATEST_TIMESTAMP server-get cost of addColumn. The only actionable items are
cross-class consistency gaps (factory row-key overload shapes and the missing
setAttribute(String, byte[]) override that its Append/Increment siblings carry),
plus one internal helper inconsistency. No functional defects in the declared
methods.

2. NAMING ISSUES
None actionable. addColumn/addColumns naming difference is load-bearing HBase
semantics and is clearly documented (AnyDelete.java:841-872 vs 713-745); keep.

3. RETURN-TYPE AND PARAMETER ISSUES
- of(Object rowKey, long timestamp, NavigableMap) | AnyDelete.java:218-221 (ctor) / 424 (factory) | Low |
  This constructor uses HBaseExecutor.toRowBytes(rowKey) whereas every other
  AnyDelete constructor uses toRowKeyBytes(rowKey) (lines 161,179,192,206). If
  the two helpers differ in null/empty/encoding handling, the same Object row key
  would convert differently through the familyMap factory than through of(Object).
  Recommendation: use toRowKeyBytes for consistency across all AnyDelete ctors.
  Confirming check: diff HBaseExecutor.toRowBytes vs toRowKeyBytes; if identical
  for all inputs this is cosmetic only.

4. MISPLACED METHODS
None. All declared methods belong on AnyDelete.

5. MISSING METHODS
- setAttribute(String, byte[]) | AnyDelete.java (absent) | Medium | target: AnyDelete |
  AnyIncrement (AnyIncrement.java:700) and AnyAppend (AnyAppend.java:635) both
  declare a byte[] overload of setAttribute that bypasses toValueBytes and makes
  byte-array intent explicit. AnyDelete (and AnyPut) lack it, so the same call
  with a byte[] literal silently routes through the Object overload here but the
  byte[] overload there. Recommendation: add setAttribute(String, byte[]) to
  AnyDelete (and AnyPut) for a uniform attribute API across all four subclasses,
  OR remove it from Increment/Append (see CROSS-CLASS NOTES — pick one direction).
- of(byte[] rowKey) and of(byte[] rowKey, int, int) | AnyDelete.java (absent) | Low | target: AnyDelete |
  AnyIncrement/AnyAppend expose dedicated byte[] row-key factories (avoid a
  conversion hop); AnyDelete routes byte[] through of(Object). Functionally fine
  (toRowKeyBytes returns byte[] as-is), so this is a consistency-only gap.
  Confirming check: verify toRowKeyBytes(byte[]) is identity — if so, behavior is
  equivalent and this is purely about API symmetry, not correctness.

6. DEPRECATION / REMOVAL CANDIDATES
None specific to AnyDelete. (Inherited compareTo is already @Deprecated in
AnyMutation.java:695-698; not re-flagged.)


################################################################################
CLASS 2 — AnyIncrement
################################################################################

1. OVERALL ASSESSMENT
AnyIncrement is a correct, well-documented wrapper. addColumn(family, qualifier,
amount) correctly models HBase's atomic counter (negative = decrement, documented
at AnyIncrement.java:418-424), and setTimeRange/getTimeRange/setReturnResults/
isReturnResults/hasFamilies/getFamilyMapOfLongs are accurate one-to-one mirrors.
Two actionable items: the equals/hashCode contract is genuinely surprising
(row-key-only equality is inherited from HBase Increment but is a real foot-gun
worth a stronger doc/override decision), and the byte[]/Object factory shape
diverges from AnyDelete/AnyPut. No functional defects in declared methods.

2. NAMING ISSUES
None actionable. addColumn(...amount) vs the sibling addColumn(...value) naming
difference correctly signals counter vs concatenation semantics; keep.

3. RETURN-TYPE AND PARAMETER ISSUES
- equals(Object) / hashCode() | AnyIncrement.java:761-774 / 728-732 | Medium |
  Delegates to HBase Increment.equals, which compares the ROW KEY ONLY — two
  increments on the same row with different columns/amounts are reported equal
  (documented at 734-759). This is inherited HBase behavior, but as a public
  equals/hashCode it violates the normal "value equality" expectation and will
  silently collapse distinct increments in a HashSet/HashMap keyed on AnyIncrement.
  Recommendation: either override to compare the full family map (preferred for a
  value-style builder) OR keep delegation but this is the only one of the four
  whose equals ignores content — call that out as deliberate. Note: AnyDelete/
  AnyAppend/AnyPut all delegate to identity-ish equals; Increment is the odd one
  because HBase Increment.equals is row-only. Treat as a documented hazard at
  minimum; an override is the cleaner fix.

4. MISPLACED METHODS
None.

5. MISSING METHODS
- of(Object, long) and of(Object, int, int) | AnyIncrement.java (absent) | Low | target: AnyIncrement |
  AnyIncrement offers of(byte[], int, int) but no Object-based offset/length or
  Object+timestamp factory, while AnyDelete/AnyPut offer the Object slice forms.
  Net effect: a non-byte[] row key cannot be sliced or timestamp-seeded at
  construction here. Low because increment timestamps are usually server-assigned.
  Confirming check: confirm no caller needs an Object-keyed timestamped increment;
  if some do, add of(Object, long).
- setReturnResults / isReturnResults absent on AnyDelete is expected (Delete has
  no such concept) — noted only to show the Increment/Append parity is correct.

6. DEPRECATION / REMOVAL CANDIDATES
None. (hashCode carries @SuppressWarnings("deprecation") at 728 because HBase
Increment.hashCode is deprecated upstream; that is an upstream artifact, not an
AnyIncrement API defect.)


################################################################################
CLASS 3 — AnyAppend
################################################################################

1. OVERALL ASSESSMENT
AnyAppend is a clear, well-documented mirror of HBase Append, with excellent
class-level guidance distinguishing byte-concatenation from Increment's counter
semantics. The actionable items are a genuine behavioral hazard inherited from
HBase — add(Cell) silently swallows a row-key mismatch instead of throwing (it
is documented, but the API still drops data silently) — and the same cross-class
factory/setAttribute shape divergence noted elsewhere.

2. NAMING ISSUES
None actionable.

3. RETURN-TYPE AND PARAMETER ISSUES
- add(Cell) | AnyAppend.java:604-608 | Medium |
  Unlike AnyDelete.add (AnyDelete.java:521, throws IOException), AnyIncrement.add
  (AnyIncrement.java:356, throws IOException), and AnyPut.add (AnyPut.java:1369,
  throws IOException), AnyAppend.add declares NO checked exception and — per its
  own Javadoc (586-587) — the wrapped Append.add() logs-and-swallows a row-key
  mismatch, so a Cell with the wrong row is silently dropped with no signal to the
  caller. This is the one place in the four-class family where a wrong-row Cell
  fails silently rather than throwing. Recommendation: validate the cell row
  against getRow() before delegating and throw IllegalArgumentException (or
  IOException, to match the siblings) on mismatch, rather than relying on the
  swallow. At minimum the divergence from the sibling add(Cell) contract should be
  removed so all four behave the same.

4. MISPLACED METHODS
None.

5. MISSING METHODS
- of(Object, long) and of(Object, int, int) | AnyAppend.java (absent) | Low | target: AnyAppend |
  Same gap as AnyIncrement: Object-keyed timestamp/slice factories are absent
  while byte[] ones exist. Append timestamp is typically server-assigned, so Low.
  Confirming check: confirm no caller constructs a timestamped Append from a
  non-byte[] key.

6. DEPRECATION / REMOVAL CANDIDATES
None active. (The commented-out add(byte[],byte[],byte[]) block at
AnyAppend.java:488-502 correctly reflects an upstream HBase deprecation and was
intentionally not exposed; leave removed.)


################################################################################
CROSS-CLASS NOTES  (AnyDelete / AnyIncrement / AnyAppend / AnyPut)
################################################################################

CC-1. setAttribute(String, byte[]) override is present on only 2 of 4.
  Present: AnyIncrement.java:700, AnyAppend.java:635.
  Absent:  AnyDelete, AnyPut (they inherit only setAttribute(String, Object)).
  Confidence: Medium. Effect: a byte[] literal at the call site binds to the
  byte[] overload on Append/Increment but to the Object overload on Delete/Put.
  Both ultimately store bytes (toValueBytes returns byte[] as-is), so behavior is
  equivalent today, but the API surface is asymmetric and the "explicit byte[]
  intent" only exists on two of the four. RECOMMENDATION: make it uniform — add
  the byte[] overload to AnyDelete and AnyPut, OR drop it from Append/Increment
  and rely on the inherited Object overload everywhere. Prefer adding to all four
  (small, type-explicit, zero behavior change).

CC-2. add(Cell) exception contract is inconsistent across the four.
  AnyDelete.add (AnyDelete.java:521), AnyIncrement.add (AnyIncrement.java:356),
  AnyPut.add (AnyPut.java:1369): all `throws IOException` and propagate a wrong-row
  mismatch. AnyAppend.add (AnyAppend.java:604): no checked exception, silently
  swallows the mismatch. Confidence: Medium. RECOMMENDATION: align AnyAppend.add
  with the siblings — detect the row mismatch and throw — so all four signal
  wrong-row Cells the same way.

CC-3. Row-key factory overload shapes diverge into two families.
  Object-keyed family (AnyDelete, AnyPut): of(Object), of(Object,long),
    of(Object,int,int), of(Object,int,int,long) [Delete] / + boolean/ByteBuffer
    variants [Put]. No byte[] factory.
  byte[]-keyed family (AnyIncrement, AnyAppend): of(Object), of(byte[]),
    of(byte[],int,int), of(byte[],long,NavigableMap). No Object slice/timestamp
    factory.
  Confidence: Low-Medium. Effect: the "natural" way to build each operation from
  a given key type differs by subclass for no semantic reason (all keys ultimately
  go through HBaseExecutor row-byte conversion). RECOMMENDATION: converge on one
  pattern — expose BOTH of(Object...) and of(byte[]...) slice/timestamp factories
  on all four (Object for ergonomics, byte[] to skip a conversion hop). Confirming
  check: confirm toRowKeyBytes/toRowBytes treat byte[] as identity so the unified
  set is behavior-preserving.

CC-4. Internal row-bytes helper is inconsistent within AnyDelete itself.
  AnyDelete uses toRowKeyBytes in 5 ctors but toRowBytes in the familyMap ctor
  (AnyDelete.java:219); AnyIncrement/AnyAppend/AnyPut use toRowBytes throughout.
  Confidence: Low. RECOMMENDATION: pick one helper for AnyDelete (toRowKeyBytes)
  and use it in all its ctors. Confirming check: diff the two helpers; if they
  agree on all inputs this is cosmetic.

CC-5. equals/hashCode semantics differ by subclass (mostly unavoidable, flag once).
  AnyDelete/AnyAppend/AnyPut delegate to HBase equals (effectively identity / not
  deep value equality). AnyIncrement delegates to Increment.equals which is
  ROW-KEY-ONLY (AnyIncrement.java:761-774). Confidence: Low (documented), but the
  row-only case is the surprising one. RECOMMENDATION: as in Class 2 §3, decide
  deliberately — either override AnyIncrement.equals to be content-aware or keep
  the inherited row-only behavior but ensure callers are warned uniformly.

Net: the three target classes are functionally sound and faithfully mirror HBase.
All actionable findings are consistency/contract-uniformity issues across the
four AnyMutation subclasses (CC-1..CC-5) plus AnyAppend.add's silent-drop hazard
(Class 3 §3 / CC-2) and AnyIncrement's row-only equals (Class 2 §3 / CC-5).
