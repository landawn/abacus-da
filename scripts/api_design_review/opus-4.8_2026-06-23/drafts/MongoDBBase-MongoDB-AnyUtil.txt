============================================================================
CLASS: MongoDBBase  (public abstract class)
============================================================================

1. OVERALL ASSESSMENT
The class is a well-organized static toolkit for BSON<->Java conversion, ID mapping, codec
configuration, and cursor/iterable extraction. The overload families (toBson/toDocument/
toBSONObject/toDBObject, each with single-arg + varargs forms; toMap; toEntity/toList; extractData
x6; stream x4) are internally consistent, null/empty edge cases are documented and mostly enforced,
and the Javadoc Usage-Examples are unusually thorough. The main design-level concerns are: a family
of four near-identical toXxx converter families whose varargs forms create a real overload-ambiguity
hazard (a single Object[] is silently treated as name/value pairs rather than one value); one
misleading Javadoc example (objectId2Filter); and a couple of return-type/parameter naming nits.
Functionality is correct; findings below are refinements, not bugs.

2. NAMING ISSUES
- toBSONObject(...) / toDBObject(...) families | MongoDBBase.java:656,718,761,826 | Low |
  Names are accurate to their return types (BasicBSONObject / BasicDBObject) and follow the toXxx
  convention used by toBson/toDocument. No rename needed; the "BSONObject" vs "DBObject" distinction
  mirrors the underlying driver types. Confirming check: return types at :656 (BasicBSONObject) and
  :761 (BasicDBObject) match the names. Listed only to record it was evaluated and is healthy.

- fromJson(String, Class<T>) / toJson(Bson) / toJson(BSONObject) / toJson(BasicDBObject) |
  MongoDBBase.java:311,356,383,413 | Low | Naming consistent with library JSON conventions
  (fromJson/toJson). Healthy. The three toJson overloads dispatch by BSON family and are a
  legitimate overload family, not duplication (verified distinct param types).

3. RETURN-TYPE AND PARAMETER ISSUES
- toBson(Object) vs toBson(Object...) ambiguity (whole toXxx varargs family: toBson, toDocument,
  toBSONObject, toDBObject) | MongoDBBase.java:447/480, 515/551, 656/718, 761/826 | High |
  The single-arg form means "convert this one object"; the varargs form means "alternating
  name/value pairs OR (length==1) a single object." Calling toDocument(new Object[]{...}) or passing
  an Object[] through a variable binds to the varargs overload and is interpreted as pairs, never as
  "one array value." This is the classic unsafe varargs-over-Object overload. Recommendation: keep
  the family but document the dispatch precedence prominently on the single-arg form too, and
  consider a distinctly-named pairs builder (e.g. toDocumentOf(Object... pairs) / a Map-returning
  AnyUtil.array2Props-style helper) so callers can opt in explicitly rather than relying on arity.
  At minimum the single-arg Javadoc should cross-warn about the array case. (Grouped: same issue for
  all four converter families — treat as ONE finding.)

- toMap(Document) / toMap(Document, IntFunction) | MongoDBBase.java:864,900 | Medium |
  toMap performs a SHALLOW copy (map.putAll(doc)); nested Documents stay Document instances, as the
  Javadoc at :854 honestly notes. The name toMap implies full Map materialization. For a tabular/DTO
  library this is a real footgun (a "Map" that still contains driver types). Recommendation: either
  rename to asMap / leave as-is but keep the explicit shallow-copy note (already present), OR add a
  deepToMap(Document) sibling for callers who need a driver-type-free structure. Confidence Medium
  because the behavior is documented; flagged anyway per ground rule 5.

- toList(MongoIterable<?>, Class<T>) return type | MongoDBBase.java:1007 | Low | Returns a mutable
  ArrayList (and in the fast path returns the very list the driver populated). Consistent with the
  library's List-returning conventions. The scalar-projection branch (size<=2, derive non-_id key)
  is subtle but documented at :1035. Healthy. Confirming check: :1009 into(new ArrayList<>()),
  :1014 returns that same list when type already matches — callers may mutate the driver-produced
  list; acceptable and conventional here.

- objectId2Filter(String) / objectId2Filter(ObjectId) | MongoDBBase.java:236,265 | Low (Javadoc
  defect, not API) | Return type Bson is correct and the two-overload family is clean. BUT the
  Usage-Example comment claims the result is {"_id": {"$oid": "..."}} whereas the implementation
  returns new Document("_id", objectId) i.e. {_id: ObjectId(...)} — the $oid extended-JSON form only
  appears if/when serialized. Recommendation: fix the example comment (out of report-only scope, but
  flag for the doc pass). Confirming check: :268 returns new Document(_ID, objectId).

4. MISPLACED METHODS (target class)
- None. Every method here is a static BSON/extraction helper shared by both MongoDB facades, which
  is exactly the documented purpose of the base. fromJson/toJson/toBson/toDocument/extractData/stream
  all belong here.

5. MISSING METHODS (target class)
- objectId2Filter has no sibling that builds a filter from a generic id VALUE of an entity, and no

  objectId2Filter(Collection<?>)/(...$in) helper would be a natural companion for delete/find-many by

  in the class.
- stream(...) family covers MongoIterable and MongoCursor but there is no stream(MongoIterable,
  selectPropNames, Class) to mirror the extractData select-props variants | MongoDBBase.java:1434-1552
  | Low | Minor symmetry gap; extractData supports column selection, stream does not. Likely

  :1509,:1550 — none take selectPropNames.

6. DEPRECATION / REMOVAL CANDIDATES
- registerIdProperty(Class, String) | MongoDBBase.java:193 (already @Deprecated) | High (confirm) |
  Correctly deprecated with a clear replacement (@Id annotation) in the Javadoc. No action beyond

- toBSONObject(...) and toDBObject(...) families target the LEGACY driver types
  (BasicBSONObject / BasicDBObject) | MongoDBBase.java:656,718,761,826 | Low | Not deprecated, but

CLASS: MongoDB  (public final class MongoDB extends MongoDBBase) — sync facade
============================================================================

1. OVERALL ASSESSMENT
A clean, minimal factory/facade: two constructors, db(), two collection(...) accessors, two
collectionExecutor(...) factories, three collectionMapper(...) factories. Naming and parameter
ordering are consistent and the Javadoc honestly documents the asymmetric validation (collection(...)
delegates validation to the driver while collectionExecutor/collectionMapper validate up front). The
notable issues are: (a) inconsistent null-validation across the collection* methods, and (b) facade
asymmetry vs the reactive twin (this class threads an AsyncExecutor; the reactive one does not).
Both are minor and largely by-design.

2. NAMING ISSUES
- db() | MongoDB.java:145 | Low | Short accessor name; consistent with the reactive twin's db()
  (reactivestreams/MongoDB.java:175), so keep for symmetry. A getDatabase() alias would read better
  but symmetry wins. Healthy.
- collection / collectionExecutor / collectionMapper | MongoDB.java:171,201,226,263,303,336,383 |
  Low | Factory-style verbs-as-nouns; consistent within the class and with the reactive twin.
  Healthy. (No of/create/valueOf expectation here — these are accessors/factories on an instance,
  not static factories.)

3. RETURN-TYPE AND PARAMETER ISSUES
- collection(String) / collection(String, Class<T>) do NO argument validation; collectionExecutor /
  collectionMapper DO (N.checkArgNotNull) | MongoDB.java:171,201 vs 226,263,303,336,383 | Medium |
  Inconsistent null-handling within one facade: collection(null) throws from deep inside the driver
  (documented at :163), while collectionExecutor(null)/collectionMapper(null,...) throw a clean
  IllegalArgumentException("collectionName"). Recommendation: add N.checkArgNotNull(collectionName,
  "collectionName") to both collection(...) overloads for a uniform, source-attributable failure.
  Confidence Medium (behavior documented, but ground rule 5 says document != excuse).
- collectionMapper(Class<T>) NPE vs IAE | MongoDB.java:303 | Low | The no-name overload throws
  NullPointerException for a null rowType (simple-name lookup dereferences first), whereas the
  two-arg overload throws IllegalArgumentException("rowType"). Inconsistent exception type for the
  same bad input across one overload family. Recommendation: validate rowType up front in the
  one-arg form so both throw IAE. Confirming check: :304 calls ClassUtil.getSimpleClassName(rowType)
  before any null check; :338 has N.checkArgNotNull(rowType,...). Same defect mirrored in the
  reactive twin (reactivestreams/MongoDB.java:373) — fix both for symmetry.

4. MISPLACED METHODS (target class)
- None. All methods are instance-level factory/accessor methods appropriate to the sync facade.

5. MISSING METHODS (target class)
- No collectionExecutor(String, Class) / collectionMapper overload that lets you pick a typed
  collection by NAME in one call without going through collection(name, type) first | MongoDB.java
  226-398 | Low | Minor; the building blocks exist (collection(name,type) + collectionExecutor(coll)).
  Confirming check: the only name-based factories are collectionExecutor(String) and
  collectionMapper(String, Class); neither lets you specify the driver collection's row type for the
  executor path. Likely intentional (executor is Document-centric). Low.

6. DEPRECATION / REMOVAL CANDIDATES
- None. No deprecated members; the commented-out pooling code (e.g. :56-58,:229-236) is dead-comment
  noise but not an API surface concern.

CLASS: AnyUtil  (public final class)
============================================================================

1. OVERALL ASSESSMENT
A one-method utility (array2Props) that turns an even-length Object[] of alternating name/value
pairs into an insertion-ordered LinkedHashMap. The method itself is correct, well-documented, and
null/odd-length edge cases are handled. The dominant finding is class PLACEMENT: AnyUtil lives in
com.landawn.abacus.da.mongodb but has ZERO mongodb callers — its only two production usages are in
aws/dynamodb/v2/DynamoDBExecutor (lines 1021, 1105). The class name is also uninformative ("AnyUtil"
says nothing about what it does).

2. NAMING ISSUES
- class AnyUtil | AnyUtil.java:13 | Medium | "AnyUtil" is a non-descriptive catch-all name; the
  class holds exactly one prop-bag helper. Recommendation: rename to something intent-revealing
  (e.g. PropsUtil) or fold the method into an existing shared util. Confirming check: the only
  member is array2Props (:70); nothing "any"-related justifies the name.
- array2Props(Object[]) | AnyUtil.java:70 | Low | Method name is reasonable and matches similar
  array2X helpers in the abacus family. Note it overlaps conceptually with MongoDBBase.toDocument(
  Object...) / toBSONObject(Object...) pair-parsing logic (same name/value-pair interpretation), but
  returns a plain LinkedHashMap rather than a BSON type, so it is not a true duplicate. Healthy name.

3. RETURN-TYPE AND PARAMETER ISSUES
- array2Props(Object[]) | AnyUtil.java:70 | Low | Returns LinkedHashMap<String,Object> (mutable,
  ordered) — appropriate. Accepts null (returns empty map) and rejects odd length with IAE. One
  asymmetry vs MongoDBBase's pair parsers: those eagerly reject a non-String NAME with IAE, whereas
  array2Props silently coerces any name via N.stringOf (documented at :39). Acceptable given the
  documented contract; just note the divergent name-handling between the two pair-parsing code paths.
  Confirming check: :82 props.put(N.stringOf(propNameAndValues[i]), ...).

4. MISPLACED METHODS (target class)
- ENTIRE CLASS is misplaced | AnyUtil.java:13 | High | Package com.landawn.abacus.da.mongodb, but
  the sole production consumers are com.landawn.abacus.da.aws.dynamodb.v2.DynamoDBExecutor:1021 and
  :1105 (verified by repo-wide grep). No mongodb class references it. Recommendation: move AnyUtil
  (or just array2Props) into the dynamodb package, or into a neutral shared util package, and update
  the two DynamoDBExecutor imports. Leaving a DynamoDB-only helper in the mongodb package is a
  cross-package coupling smell. Confirming check: grep "AnyUtil"/"array2Props" repo-wide — production
  hits only in dynamodb/v2/DynamoDBExecutor.java; remaining hits are tests.

5. MISSING METHODS (target class)
- None worth adding given the class should likely be relocated/merged rather than grown.

6. DEPRECATION / REMOVAL CANDIDATES
- AnyUtil as a standalone public class | AnyUtil.java:13 | Medium | Candidate for consolidation:
  fold array2Props into the dynamodb util that uses it (or a shared util), then remove the public
  mongodb-package class. Replacement API: the same method relocated. Confirming check: only one
  method, one production caller family (dynamodb).

CROSS-CLASS NOTES (sync MongoDB vs reactivestreams.MongoDB; conventions)
============================================================================
- FACADE SYMMETRY (mostly good): both MongoDB facades expose db(), collection(String),
  collection(String,Class), collectionExecutor(String), collectionExecutor(MongoCollection),
  collectionMapper(Class), collectionMapper(String,Class), collectionMapper(MongoCollection,Class)
  with matching names/param order. Keep them in lock-step.
- ASYMMETRY 1 — constructors: sync MongoDB has TWO ctors (MongoDatabase) and (MongoDatabase,
  AsyncExecutor) [MongoDB.java:90,118]; reactive has ONE (MongoDatabase) [reactivestreams/
  MongoDB.java:147] and no AsyncExecutor field (reactive needs none). This asymmetry is by-design
  (blocking driver needs an executor for async wrappers; reactive driver does not). No action, but
  worth recording so a future "make them symmetric" refactor does NOT add a pointless AsyncExecutor
  to the reactive facade.
- ASYMMETRY 2 — collectionMapper(Class) null handling: BOTH facades throw NPE (not IAE) for null
  rowType because getSimpleClassName runs before any null check [MongoDB.java:303-304;
  reactivestreams/MongoDB.java:373-374]. Fix in BOTH for consistency (see MongoDB §3).
- ASYMMETRY 3 — collection(String[,Class]) validation: BOTH facades skip up-front null validation on
  collection(...) while validating on collectionExecutor/collectionMapper. Consistent BETWEEN the
  twins but inconsistent WITHIN each facade. Recommend adding checkArgNotNull to both twins together.
- FACTORY NAMING: these classes use instance accessor/factory naming (db/collection/collectionExecutor
  /collectionMapper), not the static of/create/valueOf convention — correct, since they are instance
  factories, not static value factories. AnyUtil.array2Props follows the array2X helper convention.
- RETURN-TYPE CONVENTIONS: List (mutable ArrayList) from toList; abacus Stream from stream(...);
  u.Optional used downstream by mappers (per MongoDB Javadoc examples). Consistent with library norms.
  No Optional-vs-null inconsistencies found in the reviewed surface (toEntity returns null for null
  input by explicit contract, which is reasonable for a converter).
- DOC DEFECT to forward to the doc pass (not code): objectId2Filter Usage-Example claims a {$oid}
  shape that the code does not produce literally (MongoDBBase.java:236 example vs :268 impl).
