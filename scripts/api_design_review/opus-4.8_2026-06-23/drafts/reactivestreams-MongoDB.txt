4. LOW
--------------------------------------------------------------------------------
[L1] collection(String) / collection(String, Class<T>) — null-validation asymmetry vs the other factory overloads
   Signature: public MongoCollection<Document> collection(final String collectionName)            | MongoDB.java:206
              public <T> MongoCollection<T> collection(final String collectionName, Class<T>)      | MongoDB.java:240
   Confidence: Low
   Detail: collection(...) performs NO argument validation (delegates straight to mongoDatabase.getCollection),
           while the sibling collectionExecutor(String)/collectionMapper(String,...) overloads on THIS class
           do call N.checkArgNotNull(collectionName,...). So within one facade a null name is rejected with a
           framework IllegalArgumentException by some factories and with a raw driver exception (or NPE) by
           others. This exactly mirrors the blocking facade (blocking collection(String) is also unvalidated —
           blocking MongoDB.java:171), so it is a pre-existing intentional pattern, NOT a reactive-specific regression.
   Recommendation: Leave as-is for sibling symmetry, OR (if tightening) add N.checkArgNotNull to BOTH facades' collection(...) overloads together so the two siblings stay identical. Do not change only the reactive side.
   Low -> confirming check: diff the validation prologue of collection(String) here vs blocking MongoDB.java:171 — both are bare getCollection delegations; confirms this is shared-by-design, not a reactive divergence.

5. NON-ISSUES VERIFIED (explicitly checked, deliberately NOT flagged)
--------------------------------------------------------------------------------
- Missing MongoDB(MongoDatabase, AsyncExecutor) constructor: the blocking facade has a 2-arg ctor (MongoDB.java:118)
  because its blocking MongoCollectionExecutor wraps sync driver calls in an AsyncExecutor. The reactive driver is
  natively non-blocking, the reactive MongoCollectionExecutor ctor takes NO AsyncExecutor (MongoCollectionExecutor.java:181),
  and this class holds no asyncExecutor field. The single-arg ctor is therefore CORRECT, not a missing overload.
- collectionExecutor / collectionMapper accessor naming (coll(), collectionExecutor()) referenced in Javadoc:
  verified to exist on the reactive collaborators (MongoCollectionExecutor.java:216, MongoCollectionMapper.java:220).
- protected static readRow(Document, Class<T>) bridge (line 502): mirrors the package-bridge pattern; the blocking
  facade does not declare it but the blocking package has its own collaborators — not an asymmetry worth flagging.
- No factory method present on the blocking MongoDB is absent here: the full {db, collection x2, collectionExecutor x2,
  collectionMapper x3} set is mirrored 1:1.
- Javadoc "Reactive Streams Publisher, not abacus Stream" disclaimers are accurate and helpful; no over-claim of abacus Stream return types.

================================================================================
6. CROSS-CLASS NOTES (symmetry with blocking MongoDB; shared base usage)
--------------------------------------------------------------------------------
- Factory symmetry is EXCELLENT: identical method names, identical parameter ordering, identical overload families
  (String / MongoCollection<Document> / Class) versus the blocking facade. Return types differ only by reactive vs
  blocking executor/mapper, which is the intended contract difference.
- The ONLY intentional structural delta is the AsyncExecutor constructor + field, correctly absent here (reactive
  driver is natively async). This is a feature, not a gap.
- Both siblings share MongoDBBase statics/protected helpers and call the same readRow/toEntity bridge for ID handling,
  keeping _id mapping consistent across blocking and reactive paths. No inherited-base re-flag warranted.
- The two LOW items (L1 validation asymmetry on collection(...), L2 NPE-vs-IAE on collectionMapper(Class)) are
  PRE-EXISTING and SHARED with the blocking facade. If addressed, fix both siblings together to preserve symmetry;
  fixing only the reactive side would INTRODUCE a cross-sibling inconsistency.

OVERALL: This reactive facade is a clean, faithful mirror of the blocking MongoDB. No actionable reactive-specific
defects; the two LOW findings are shared cosmetic/consistency nits inherited from the blocking design.
