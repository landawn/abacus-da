API DESIGN REVIEW — HBaseExecutor (public final class, AutoCloseable)
File: abacus-da-all/src/main/java/com/landawn/abacus/da/hbase/HBaseExecutor.java (~3886 lines)
Reviewer pass date: 2026-06-23

================================================================================
1. OVERALL ASSESSMENT
================================================================================
HBaseExecutor is a well-structured, mature sync facade. The CRUD/scan/coprocessor
families are clean, consistently overloaded across the four input shapes (native
HBase type, List/Collection of native type, AnyXxx, Collection<AnyXxx>), and well
documented with accurate null/empty/exception edge-case notes. Resource ownership
(short-lived Table closed in finally; lazy scan Stream that owns its Table+Scanner)
is correct and clearly stated. The class is symmetric with AsyncHBaseExecutor at the
operation level, and the inner HBaseMapper is a thin, correct typed wrapper. The
actionable findings are modest and mostly about consistency, not correctness: the
public get/scan overloads name their conversion-target parameter `targetType` while
the async mirror, the static helpers in this same class, and the rest of the family
use `targetClass`; the static result-conversion helpers (toList/toEntity) arguably
belong on a dedicated mapping util rather than on the executor; a few sibling-gap and
deprecation items round out the list. No behavioral bugs were found in the public API.

================================================================================
2. NAMING ISSUES
================================================================================

[N1] get(...) and scan(...) conversion overloads name the Class param `targetType`
     Family: get(String, Get, Class<T>) / get(String, List<Get>, Class<T>) /
             get(String, AnyGet, Class<T>) / get(String, Collection<AnyGet>, Class<T>);
             scan(String, String, Class<T>) and the other 5 typed scan overloads.
     HBaseExecutor.java:1713, 1743, 1770, 1801, 2034, 2060, 2084, 2110, 2137, 2170
     Confidence: High
     Recommendation: Rename the parameter from `targetType` to `targetClass` across
     all 10 public typed overloads. It is a `Class<T>`, not a Type<T>; the static
     helpers in the SAME class (toList/toEntity/toValue at :574, :610, :731, :759)
     and EVERY corresponding AsyncHBaseExecutor overload (:512, :550, :586, :624,
     :920, :966, :1011, :1058, :1101, :1145) already use `targetClass`. This is a
     pure parameter-name change (no signature/erasure impact).

[N2] incrementColumnValue(...) — "ColumnValue" verbosity vs the rest of the API
     Family: incrementColumnValue(String, Object, String, String, long[, Durability])
             and its byte[] variants.
     HBaseExecutor.java:2609, 2637, 2663, 2700
     Confidence: Low
     Recommendation: Name is inherited verbatim from the HBase Table API
     (Table.incrementColumnValue), so keeping it maximizes familiarity; do NOT rename.
     Flagged only to record it was considered and intentionally left.
     Confirming check: org.apache.hadoop.hbase.client.Table#incrementColumnValue has
     the identical name — by-design pass-through, leave as-is.

3. RETURN-TYPE AND PARAMETER ISSUES
================================================================================

[R1] get(String, AnyGet/Get, Class<T>) returns bare T (null/default) — no Optional
     Family: get(String, Get, Class<T>) :1713; get(String, AnyGet, Class<T>) :1770
             (and HBaseMapper.get(K) :3222 / get(AnyGet) :3411).
     Confidence: Low
     Recommendation: KEEP returning T (null/default for a missing row). The whole
     library uses null-for-single-get rather than Optional here, and the raw
     get(...)->Result overloads return a present-but-empty Result (never null), which
     is the natural HBase contract. An Optional variant would be inconsistent with the
     family and with Async (which returns ContinuableFuture<T>, not <Optional<T>>).
     No change; recorded so the Optional-vs-null question is explicitly resolved.
     Confirming check: AsyncHBaseExecutor.get(...,Class) returns ContinuableFuture<T>;
     no Optional anywhere in either executor — consistent by design.

[R2] List-of-Get get/exists return positionally-aligned lists, but the typed
     get(...,Class) overloads silently DROP empty rows (non-aligned)
     Family: get(String, List<Get>, Class<T>) :1743;
             get(String, Collection<AnyGet>, Class<T>) :1801
             (vs the raw get(String, List<Get>) :1609 which IS positionally aligned).
     Confidence: Medium
     Recommendation: This asymmetry is real and already documented (the typed overloads
     skip empty Results via toList(List<Result>) at :656/:668, so the result list is
     shorter and not index-aligned, whereas the raw overloads keep empty Results in
     place). It is defensible (you usually do not want null entities in a typed list)
     but it is a genuine consistency trap. Either (a) keep as-is and accept the
     documented behavior, or (b) add a clearly-named variant that preserves alignment.
     Lowest-risk: leave behavior, but make the divergence prominent at the family level
     (it is currently only in the @return text). No signature change recommended.

[R3] scan(...) "scan whole table" requires the AnyScan path; no family-less convenience
     Family: scan(String, String|byte[][, qualifier]) :1826/1850/1871/1894 and typed
             counterparts; the only "all rows/all families" entry is
             scan(String, AnyScan) with AnyScan.create().
     Confidence: Low
     Recommendation: Do NOT add a bare scan(String tableName) / scan(String, Class<T>).
     A family-less full-table scan is already expressible as scan(tableName,
     AnyScan.create()) / scan(tableName, AnyScan.create(), targetClass), and an
     argument-light "scan everything" overload would be an easy footgun on large
     tables. Recorded as considered-and-declined.
     Confirming check: scan(String, Scan) at :1962 accepts a raw `new Scan()` (no
     family) — full-table scan is reachable; no gap.

4. MISPLACED METHODS (with target class)
================================================================================

[M1] Static Result→entity conversion helpers live on the executor
     Family: public static <T> List<T> toList(ResultScanner, Class<T>) :574;
             public static <T> List<T> toList(ResultScanner, int, int, Class<T>) :610;
             public static <T> T toEntity(Result, Class<T>) :731.
     Confidence: Low
     Recommendation: These are pure, connection-independent mapping utilities (they take
     a ResultScanner/Result the caller already owns and need no Connection/Admin). They
     would sit more naturally on a stateless mapping util (e.g. an HBaseMappers / Hbase
     util type) than on the connection-bound executor, mirroring how N.registerConverter
     is wired in the static initializer. However they are genuinely useful next to the
     executor and are public API in wide use; moving them is a real break for marginal
     gain. Recommend KEEP on HBaseExecutor and treat as by-design; do not relocate.
     Confirming check: toList(ResultScanner,...) / toEntity(Result,...) reference no
     instance state — they are static and Connection-free, so placement is a style
     choice, not a correctness issue.

5. MISSING METHODS (with target class)
================================================================================

[X1] No Result→Map conversion (commented-out toMap exists)
     Target: HBaseExecutor (static), sibling to toEntity(Result, Class<T>) :731.
     Evidence: dead commented-out toMap(Result) / toMap(Result, IntFunction) at
     HBaseExecutor.java:735-741; toValue/toEntity explicitly REJECT Map targets
     (IllegalArgumentException "Map type is not supported", :771 and :816).
     Confidence: Medium
     Recommendation: Either delete the commented-out block (it advertises an API that
     was deliberately disabled) or, if a column→value Map view is genuinely wanted,
     add a documented toMap(Result) returning Map<String,Object> keyed by
     family:qualifier. Minimum action: remove the dead comment so the public surface
     does not imply a planned-but-missing method. (Report-only; no code change here.)

[X2] mapper(Class<T>) and mapper(Class<T>, String, NamingPolicy) but no
     mapper(Class<T>, String) CAMEL_CASE-default overload
     Family: mapper(Class) :1327; mapper(Class, String, NamingPolicy) :1379.
     Confidence: Low
     Recommendation: The common case "I have no @Table annotation but want the default
     CAMEL_CASE policy" forces callers to pass NamingPolicy.CAMEL_CASE explicitly. A
     mapper(Class<T>, String tableName) overload defaulting to CAMEL_CASE would round
     out the family and match the implicit default used by mapper(Class). Low value,
     low risk — add only if convenience is prioritized.
     Confirming check: mapper(Class) :1339 hardcodes NamingPolicy.CAMEL_CASE as its
     default, so a 2-arg (Class,String) overload would be a consistent, non-surprising
     addition.

[X3] No put/delete returning anything; mutateRow returns void — fine. No actionable
     primitive/variant gap found among append/increment/incrementColumnValue siblings.
     Confidence: High (negative finding) — append/increment both return Result;
     incrementColumnValue returns long for both String and byte[] variants, each with a
     Durability overload. The family is complete and symmetric. No action.

CROSS-CLASS NOTES (vs AsyncHBaseExecutor and executor conventions)
================================================================================
- Operation-level symmetry with AsyncHBaseExecutor is EXCELLENT: every public
  exists/get/scan/put/delete/mutateRow/append/increment/incrementColumnValue/
  coprocessorService/batchCoprocessorService overload has a 1:1 async mirror
  (AsyncHBaseExecutor.java:191-2076). Async wraps each in ContinuableFuture<...> and
  uses ContinuableFuture<Void> where sync is void — the correct convention. The
  sync-only members (admin/connection/async/getTable/mapper/registerRowKeyProperty and
  the static toList/toEntity) are appropriately NOT mirrored.

- NAMING DRIFT [N1] is the one concrete cross-class inconsistency: sync get/scan typed
  overloads use `targetType`; the async mirror and this class's own static helpers use
  `targetClass`. Align the sync side to `targetClass`. This is the single highest-value,
  lowest-risk fix and should also be checked against AsyncHBaseExecutor staying the
  authority (it already uses targetClass).

- The four package-private Object-rowKey convenience overloads
  (exists(String,Object) :1389, get(String,Object) :1548, get(String,Object,Class) :1679,
  delete(String,Object) :2293) are deliberately NON-public (comment at :1387-1388 /
  :1546-1547 / :1677-1678 / :2291-2292 explains the Object-ambiguity rationale). They
  are out of public-API scope; AsyncHBaseExecutor does not expose them either —
  consistent. No action; noted so they are not mistaken for missing public overloads.

- HBaseMapper inner class faithfully delegates to the outer executor with the bound
  table name and entity class; its method set is a correct subset and its parameter
  naming on incrementColumnValue matches the outer class. No independent findings.
