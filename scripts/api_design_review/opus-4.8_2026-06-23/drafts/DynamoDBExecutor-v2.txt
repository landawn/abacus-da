============================================================
1. FUNCTIONALITY
============================================================
No functional/correctness bugs found. The query/list/scan/stream auto-pagination loops
(list:2977-2987, query:3098-3110, stream:3209-3250, scan:3443-3485) were traced and are
correct: the first `newQueryRequest`/`newScanRequest` assignment is always overwritten inside
the do/while before being used, so no page is skipped or double-read. The package-private
internal converters (toValue/readRow/createRowMapper/extractData) are out of public scope and
behave consistently with the v1 twin (incl. the deliberate empty-string `isNotEmpty`-then-
`!=null` fallback in toValue — present in both versions, intentional).

(no actionable findings)

  | DynamoDBExecutor.java:442,498,556 | High
  | v1 twin names the identical helpers toAttributeValue / toAttributeValueUpdate
    (v1 DynamoDBExecutor.java:906,949,982). v2 renamed them to attrValueOf / attrValueUpdateOf.

4. PARAMETERS
============================================================

query(QueryRequest, Class<?>)  vs  list / stream(QueryRequest, Class<T>)
  | DynamoDBExecutor.java:3093 | Medium
  | query(QueryRequest, Class) deliberately ACCEPTS a null targetClass (line 3094:
    `if (targetClass == null || Map.class.isAssignableFrom(targetClass))`) and its Javadoc
    documents "or {@code null}/a Map type". But the sibling generic overloads list(QueryRequest,
    Class<T>) and stream(QueryRequest, Class<T>) declare targetClass "Must not be null" and have
    no null short-circuit. Inconsistent null-acceptance within the same query overload family.
    Recommend either (a) document & support null uniformly (treat null as Map/raw across list/
    stream too), or (b) make query() reject null for symmetry. Prefer (a) since query() already
    relies on it.

asKey(Object...) / asItem(Object...) / asUpdateItem(Object...)
  | DynamoDBExecutor.java:708,812,932 | Low
  | asKey(Object...) just delegates to asItem(Object...); the only difference from asItem is
    intent/naming. The Javadoc already warns of the misuse risk (683, 798). Same as v1 — keep,
    but consider @Beta or a note that asKey(Object...) is purely a semantic alias. No change
    required; consistent with v1.
  | Low: confirm asKey(Object...) body == asItem(a) (it does, line 709).

6. MISSING METHODS / DEPRECATION
============================================================

async()  — MISSING on v2 sync (present on v1 sync)
  | v1 DynamoDBExecutor.java:505 ; absent in v2 | Medium
  | v1 sync exposes `public AsyncDynamoDBExecutor async()` as a bridge to its async twin. The
    v2 sync executor provides NO bridge to aws/dynamodb/v2/AsyncDynamoDBExecutor (which exists
    and is independently constructed from a DynamoDbAsyncClient). This is a genuine API gap, not
    an SDK limitation: the async twin exists. EITHER add a parallel accessor (note: v2 sync holds
    a sync DynamoDbClient and the async twin needs a DynamoDbAsyncClient, so a no-arg async()
    cannot share the client — an `async(DynamoDbAsyncClient)` factory or a doc note "construct
    AsyncDynamoDBExecutor directly" would resolve the asymmetry), OR document the intentional
    divergence in the class Javadoc's "Migration from v1" section.
  | Low: confirm no async()/async-accessor anywhere in v2 sync (grep: none).

toMap(Object[] propNameAndValues)  — MISSING on v2 (present on v1)
  | v1 DynamoDBExecutor.java:1218 ; absent in v2 | Low
  | v1 exposes public static toMap(Object[]) (alternating name/value pairs -> LinkedHashMap,
    String.valueOf on keys). v2 only has toMap(Map) and toMap(Map,IntFunction). Minor: the
    Object[] variant is a generic util only loosely tied to DynamoDB and a caller can use
    N.asMap; likely an intentional trim. Recommend either restore for v1 parity or confirm the

CROSS-CLASS NOTES
============================================================

V1 vs V2 DIVERGENCE
- Helper RENAME (accidental drift): v1 toAttributeValue/toAttributeValueUpdate -> v2
  attrValueOf/attrValueUpdateOf. Same semantics, different name across twins. (Section 2)
- BRIDGE ASYMMETRY: v1 sync has async() -> AsyncDynamoDBExecutor; v2 sync has no equivalent
  bridge even though aws/dynamodb/v2/AsyncDynamoDBExecutor exists. (Section 6)
- toMap(Object[]) present v1, dropped v2. (Section 6)
- By-design SDK gaps (NOT findings): no DynamoDBMapper accessor, no config/AsyncExecutor
  constructors in v2 — SDK v2 uses DynamoDbEnhancedClient and a different async model.
- Public-method SHAPE between v1 and v2 sync is otherwise 1:1 (getItem/batchGetItem/putItem/
  batchWriteItem/updateItem/deleteItem/list/query/stream/scan families, Filters, ConditionBuilder,
  Mapper) — only the *Result -> *Response return-type swap (PutItemResult->PutItemResponse etc.),
  which is the expected SDK-v2 type rename, not drift.

LIBRARY CONVENTION DEVIATIONS
- get=Optional/gett=null: the executor returns bare null from getItem (Section 3). Consistent
  across the whole DynamoDB family and v1; treated as documented-by-design, not flagged.
- @Beta: no @Beta annotations anywhere in this class. asKey(Object...)/asItem(Object...) and the
  ConditionBuilder fluent API are the experimental-feeling surfaces; consider @Beta on the
  Object... convenience overloads if the project applies @Beta elsewhere. Low priority.
- ASYNC RETURN TYPE (belongs to the async-twin review, noted here for the pair): the v2 async
  twin returns java.util.concurrent.CompletableFuture, NOT the library-standard
  ContinuableFuture used by every other async executor (HBase/Cassandra/v1-DynamoDB async all
  use ContinuableFuture; v1's async() Javadoc here even advertises ContinuableFuture at line 496).
  This is the single largest convention deviation in the v2 DynamoDB pair — flag it on
  AsyncDynamoDBExecutor(v2).

AnyUtil COUPLING
- Lines 1021 & 1105 call mongodb/AnyUtil.array2Props; this class is the sole main-source user.
  Already filed; do not re-derive. (Section 5)
