RELATIONSHIP
This is the Reactive Streams (Project Reactor Mono/Flux) twin of the blocking MongoCollectionMapper<T>.
Both are thin, type-bound wrappers that delegate every call to their respective MongoCollectionExecutor,
substituting rowType for the per-call Class<T>. The reactive class is a faithful mirror: read scalars ->
Mono<wrapper> (empty == absent), single doc reads -> Mono<T>, multi-doc reads -> Flux<T>, query -> Mono<Dataset>,
writes -> Mono<Result>. The mapping discipline is consistent and correct across the surface. Findings below are
mostly low-severity drift; the only substantive ones are a documented distinct() decoding hazard inherited from
the executor and a few naming/return-shape divergences from the blocking sibling.

================================================================================
1. FUNCTIONALITY (clarity / overlap / reactive return-type correctness)
================================================================================

distinct(String) | reactivestreams/MongoCollectionMapper.java:3016 ; distinct(String,Bson):3053 | Medium |
  Reactive distinct() delegates to collectionExecutor.distinct(fieldName, rowType) which (executor :3697 /
  :3725) calls the NATIVE driver coll.distinct(fieldName, rowType). The blocking sibling deliberately does NOT
  do this: blocking distinct() (sync mapper :3112-3133) routes through a hand-built $group/$project pipeline
  (distinctPipeline) precisely because — per its own code comment (sync :3118-3121) — "the driver's native
  distinct(fieldName, entityClass) decodes each raw VALUE with the entity codec and throws
  BsonInvalidOperationException for any scalar field." So for the normal case (T is a POJO entity, fieldName is
  a scalar field), reactive userMapper.distinct("country") will error on subscription while the blocking twin
  succeeds. The reactive Javadoc papers over this (:2974-3015 tells users to call the executor for scalars),
  but per ground rule 5 a documented inconsistency is still a finding: the two mappers with the same name and
  signature behave differently for the same input. Recommendation: make reactive distinct() mirror the blocking
  pipeline approach (route through aggregate(distinctPipeline(...))) so the mapper-level contract is identical
  across blocking/reactive, OR, if the divergence is intended, drop the entity-typed mapper distinct() overloads
  entirely and force callers to the executor's typed distinct(field, Class). Current state is an inconsistent
  twin. (Low->confirm: run reactive distinct("aScalarField") against a POJO mapper; expect onError
  BsonInvalidOperationException.)

CROSS-CLASS NOTES (blocking-vs-reactive divergence; mapper-vs-executor surface)
================================================================================
- distinct() semantics differ between the twins (§1): blocking routes scalar distinct through an aggregation
  pipeline and works; reactive delegates to native driver distinct and will onError for scalar fields on a POJO
  mapper. This is the one behavioral (not just cosmetic) divergence and the highest-value fix. Root cause lives
  in reactivestreams/MongoCollectionExecutor.distinct (:3694, :3721) — fixing it there (or rerouting the mapper)
  realigns both layers at once.
- Accessor name drift: mapper exposes collectionExecutor() (reactive) vs mongoCollectionExecutor() (blocking);
  executor exposes coll() (reactive) — naming is internally consistent within reactive but inconsistent with the
  blocking mapper. Cosmetic.
- Scalar-read Nullable->Mono collapse (§3) is an intrinsic, well-documented reactive limitation, not a defect.
- Everything else is a clean, faithful mirror; no extra/missing public ops, consistent parameter ordering,
  correct cold-publisher semantics documented throughout.
