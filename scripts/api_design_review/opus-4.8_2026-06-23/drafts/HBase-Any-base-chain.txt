API DESIGN REVIEW — HBase "Any*" base chain
Targets: AnyOperation, AnyOperationWithAttributes, AnyQuery, AnyMutation
(public methods inherited by AnyGet / AnyScan / AnyPut / AnyDelete / AnyIncrement / AnyAppend)
Reviewer model: opus-4.8   Date: 2026-06-23   Mode: REPORT-ONLY

================================================================================
1. OVERALL ASSESSMENT
================================================================================
These four package-private abstract bases are thin, faithful fluent wrappers over
the Apache HBase client hierarchy (Operation -> OperationWithAttributes -> Query /
Mutation), and they are in very good shape. Verified against hbase-client 2.5.6:
method names, overload families, return types, and the by-design duplication of
getACL/setACL across AnyQuery and AnyMutation (HBase declares these on Query and
Mutation separately, NOT on the shared OperationWithAttributes base) all correctly
mirror HBase — none of that is a defect. The Javadoc is unusually thorough and its
factual claims check out (e.g. getAttributesMap() is documented "read-only", and
HBase indeed returns Collections.unmodifiableMap). The fluent setters correctly
narrow HBase's base-typed returns (e.g. setId/setPriority) to the concrete AO/AOWA/
AQ/AM type, which is a genuine ergonomic improvement over raw HBase. compareTo(Row)
is deprecated consistently here and in the AnyGet override. Only a small number of
actionable items remain, and all are consistency-level (parameter naming drift in
one overridden family; one exception-handling asymmetry; a couple of low-confidence
return-type/mutability notes). No misplaced methods, no missing-method gaps of
note, and no removal candidates beyond the already-deprecated compareTo.

================================================================================
2. NAMING ISSUES
================================================================================

N1. setColumnFamilyTimeRange(String, long, long) / setColumnFamilyTimeRange(byte[], long, long)
    AnyQuery.java:504, AnyQuery.java:545  (overridden in AnyScan.java:674, AnyScan.java:708)
    Confidence: Medium
    Parameter names drift across the override. AnyQuery names them
    (cf, minStamp, maxStamp); AnyScan's @Override names them
    (family, minTimestamp, maxTimestamp). Same method, two different public
    parameter-name vocabularies — confusing in IDE hints / generated docs.
    Recommendation: pick ONE vocabulary for the whole family. Prefer
    (family, minStamp, maxStamp): "family" reads better than the abbreviation
    "cf" (and matches addFamily/addColumn elsewhere), while "minStamp/maxStamp"
    matches HBase's own setColumnFamilyTimeRange signature and the half-open
    [minStamp, maxStamp) wording already used in the Javadoc. Align AnyQuery and
    AnyScan to the chosen names.

    (No other naming defects. getFingerprint/toMap/toJson/toString, getId/setId,
    getPriority/setPriority, get/has, getCacheBlocks-style names, etc. either
    mirror HBase verbatim or already follow the library's lower-camel toJson
    convention — see section 6 note on toJSON.)

================================================================================
3. RETURN-TYPE AND PARAMETER ISSUES
================================================================================

R1. toJson() / toJson(int)                         AnyOperation.java:128, 147
    getAuthorizations()                            AnyQuery.java:175
    getCellVisibility()                            AnyMutation.java:292
    Confidence: Medium
    Exception-handling is asymmetric across the wrapper. toJson wraps HBase's
    checked IOException into the library's UncheckedIOException (good — hides the
    leaked checked exception). But getAuthorizations() and getCellVisibility()
    still propagate HBase's checked DeserializationException straight through.
    A wrapper whose stated value is "a simpler API" should be consistent: either
    all three hide the checked exception or none do.
    Recommendation: wrap DeserializationException into an unchecked library
    exception (mirroring the toJson/UncheckedIOException pattern) on
    getAuthorizations() and getCellVisibility() so no HBase checked exception
    leaks through the fluent API. (Keep, if a deliberate decision was made to
    surface deserialization failures as checked — but then document why toJson
    differs.)

R2. getFamilyCellMap()                             AnyMutation.java:191
    Confidence: Low
    Returns the LIVE, mutable NavigableMap<byte[],List<Cell>> backing the
    mutation (Javadoc admits "mutating it changes the mutation"), whereas the
    sibling accessor getAttributesMap() (AnyOperationWithAttributes.java:115)
    returns an unmodifiable view. This mutable/immutable inconsistency between
    two "get the backing map" accessors is a minor surprise.
    This mirrors HBase exactly (HBase's getFamilyCellMap is also live), so it is
    likely intentional for batch/coprocessor callers. Recommendation: keep, but
    consider strengthening the Javadoc warning (it already notes the risk).
    Confirming check: confirm no internal caller relies on mutating the returned
    map; if none, an unmodifiable wrapper would be safer but would diverge from
    HBase semantics.

R3. get(String, String) / get(byte[], byte[])      AnyMutation.java:439, 453
    Confidence: Low
    Returns List<Cell>; method name "get" is HBase's (Mutation.get). In a
    library that elsewhere favors Stream/immutable returns, a List of HBase Cells
    from a method named simply "get" reads ambiguously next to the Optional-style
    accessors. Mirrors HBase, so likely keep.
    Recommendation: keep (faithful HBase mirror). Confirming check: scan sibling
    Any* concrete classes for a "getCellList"/"cells" naming convention; if one
    exists, consider an aligned alias.

================================================================================
4. MISPLACED METHODS (with target class)
================================================================================
None. Notably, getACL/setACL appearing in BOTH AnyQuery and AnyMutation is NOT a
misplacement and must not be hoisted to AnyOperationWithAttributes: verified
against hbase-client 2.5.6 that HBase declares getACL/setACL separately on Query
and Mutation (they are absent from OperationWithAttributes). The wrapper correctly
mirrors that split.

================================================================================
5. MISSING METHODS (with target class)
================================================================================
None actionable. The bases cover the full public surface of their HBase
counterparts that is meaningful for a fluent client wrapper. (extraHeapSize /
getAttributeSize on the HBase side are protected and intentionally not surfaced.)

================================================================================
6. DEPRECATION / REMOVAL CANDIDATES (with replacement APIs)
================================================================================

D1. compareTo(Row)                                 AnyMutation.java:697
    Confidence: High (informational — already handled correctly)
    Already @Deprecated with the correct replacement note (use Row.COMPARATOR);
    the AnyGet override (AnyGet.java:992) is deprecated consistently. No action
    needed beyond keeping the deprecation. Listed only to confirm it was reviewed.

    Note (not a finding): the previously-existing toJSON()/toJSON(int) and
    setFamilyCellMap()/getTimeStamp() are already removed (commented out) in favor
    of toJson()/toJson(int) and the HBase-2.x replacements — the dead commented
    blocks (AnyOperation.java:155-176, AnyMutation.java:195-220) could be deleted
    for cleanliness but carry no API impact.

================================================================================
CROSS-CLASS NOTES (consistency vs sibling Any* classes)
================================================================================
- Parameter-name drift (finding N1): AnyQuery base uses cf/minStamp/maxStamp;
  the AnyScan override uses family/minTimestamp/maxTimestamp for the SAME
  setColumnFamilyTimeRange family. Align both (and AnyGet inherits the base
  version, so fixing AnyQuery propagates there).
- Exception asymmetry (finding R1) is a hierarchy-wide consistency issue:
  AnyOperation hides IOException but AnyQuery/AnyMutation expose
  DeserializationException.
- Healthy mirroring confirmed across the family: getACL/setACL split
  (Query vs Mutation), getFingerprint/toMap/toJson/toString (Operation),
  has(...)/get(...) overload ladders (Mutation), and the AO/AOWA/AQ/AM self-type
  fluent returns are all consistent and correct. No duplicate-method defects
  arise from inheritance.
