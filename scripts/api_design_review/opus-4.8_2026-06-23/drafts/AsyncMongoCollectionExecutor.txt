API DESIGN REVIEW — AsyncMongoCollectionExecutor
Target: abacus-da-all/src/main/java/com/landawn/abacus/da/mongodb/AsyncMongoCollectionExecutor.java
Sync sibling: abacus-da-all/src/main/java/com/landawn/abacus/da/mongodb/MongoCollectionExecutor.java
Reviewer: opus-4.8  Date: 2026-06-23
Scope: all PUBLIC methods (incl. deprecated/beta); overload families treated as one unit.

RELATIONSHIP ESTABLISHED
------------------------
AsyncMongoCollectionExecutor is the async twin of MongoCollectionExecutor (same package). It is a thin
wrapper: every method submits the equivalent blocking MongoCollectionExecutor call to a backing
AsyncExecutor and returns a ContinuableFuture<T> (or ContinuableFuture<Void> for void sync ops).
The convention is correct and consistent: Optional/Nullable/OptionalXxx payloads are preserved INSIDE
the future (never collapsed to null), matching the sync return shapes. Construction is package-private;
obtained via MongoCollectionExecutor.async(). This mirrors the HBase async pattern (ContinuableFuture).
A `sync()` accessor exists (line 166), symmetric with sync's `async()` (line 232).

Because this is a mirror class, a method present in both classes is NOT a duplicate. Findings below are
driven by (a) missing async wrappers for sync operations, and (b) annotation-mirroring inconsistencies.


=== 1. OVERALL ===
Solid, faithful async mirror of the sync executor. Return-type discipline is good throughout
(ContinuableFuture<Optional<T>>, ContinuableFuture<Nullable<T>>, ContinuableFuture<OptionalXxx>,
ContinuableFuture<Void> for void ops). Parameter ordering matches the sync sibling in every family that
exists on both. No per-method Executor params (the AsyncExecutor is held at construction) — this is
consistent across the whole class, no inconsistency there. The actionable issues are a small set of
SYNC operations that have NO async wrapper, plus an @Beta-annotation mismatch on the queryForXxx family.
No incorrectly-typed or mis-ordered async wrappers were found.


=== 2. NAMING ===
No naming findings. Async method names mirror the sync sibling exactly (exists/count/get/gett/findFirst/
list/queryForXxx/queryForSingleValue/queryForSingleNonNull/query/stream/watch/insertOne/insertMany/
updateOne/updateMany/replaceOne/deleteOne/deleteMany/bulkInsert/bulkWrite/findOneAndUpdate/
findOneAndReplace/findOneAndDelete/distinct/aggregate/groupBy/groupByAndCount/mapReduce). The `gett`
nullable-variant naming is intentionally preserved from the sync class.


=== 3. RETURN-TYPE / PARAMETER ===
3a. queryForXxx primitive family — @Beta present on SYNC, ABSENT on ASYNC twin.
    Family: queryForBoolean/Char/Byte/Short/Int/Long/Float/Double(String,Bson),
            queryForString/queryForDate(String,Bson), queryForDate(String,Bson,Class),
            queryForSingleValue(String,Bson,Class)
    | AsyncMongoCollectionExecutor.java:1078,1114,1150,1186,1225,1261,1298,1335,1371,1409,1449,1490
    | Medium
    | The sync twins are all annotated @Beta (MongoCollectionExecutor.java:1213,1247,1281,1317,1351,1385,
      1419,1453,1487,1520 and the queryForDate(…,Class)/queryForSingleValue line just below). The async
      wrappers carry NO @Beta. A mirror class should mirror stability annotations so both views expose the
      same API-stability contract; right now the async surface silently presents Beta operations as stable.
      Recommendation: add @Beta to the async queryForXxx family to match the sync sibling (or, if these are
      considered stable now, remove @Beta from the sync side — but symmetry is the goal).
      NOTE: queryForSingleNonNull(String,Bson,Class) is NOT @Beta on sync (line 1656 has no @Beta), so the
      async one at :1533 correctly has none — exclude it from this fix.

No other return-type or parameter findings: every async overload's parameter list and generic signature
matches its sync counterpart; OptionalXxx/Nullable/Optional payloads are correctly nested inside the
future rather than flattened to null.


=== 4. MISPLACED METHODS ===
None. All operations belong on the collection-level async executor.


=== 5. MISSING METHODS (sync ops with NO async wrapper) ===
Grouped: async lacks the following operations that exist on MongoCollectionExecutor. Each is an
implied-overload / operation gap, not by-design (the wrapper is mechanical and cheap).

5a. stream() and stream(Class<T>) — unfiltered full-collection stream overloads.
    Sync: MongoCollectionExecutor.java:2041 (stream()), :2084 (stream(Class))
    Async present: only stream(Bson...) and richer variants (AsyncMongoCollectionExecutor.java:1807+).
    | AsyncMongoCollectionExecutor.java (absent; nearest family at :1807)
    | High
    | Add ContinuableFuture<Stream<Document>> stream() and <T> ContinuableFuture<Stream<T>> stream(Class<T>)
      delegating to collectionExecutor.stream()/stream(rowType). The async class begins its stream family
      at stream(Bson filter), dropping the two no-filter entry points the sync class offers.

5b. estimatedDocumentCount() and estimatedDocumentCount(EstimatedDocumentCountOptions).
    Sync: MongoCollectionExecutor.java:415, :438
    | AsyncMongoCollectionExecutor.java (absent; count family at :253-300)
    | Medium
    | Add ContinuableFuture<Long> estimatedDocumentCount() and an overload taking
      EstimatedDocumentCountOptions, delegating to the sync methods. Async exposes count()/count(Bson)/
      count(Bson,CountOptions) but omits the fast metadata-count path entirely. (NOTE: the async file does
      not even import EstimatedDocumentCountOptions, confirming the gap is an omission, not a deliberate cut.)

5c. groupBy(String,Class<T>) and groupBy(Collection<String>,Class<T>) — typed group overloads.
    Sync: MongoCollectionExecutor.java:4255, :4304 (both @Beta)
    Async present: only groupBy(String) :3561 and groupBy(Collection) :3586 (Document-typed).
    | AsyncMongoCollectionExecutor.java (absent; group family at :3561-3588)
    | Medium
    | Add <T> ContinuableFuture<Stream<T>> groupBy(String,Class<T>) and
      <T> ContinuableFuture<Stream<T>> groupBy(Collection<String>,Class<T>) (mark @Beta to match sync).
      Async users can only get Document-typed groupings; the sync typed-conversion convenience is lost.

5d. groupByAndCount(String,Class<T>) and groupByAndCount(Collection<String>,Class<T>) — typed variants.
    Sync: MongoCollectionExecutor.java:4358, :4406 (both @Beta)
    Async present: only groupByAndCount(String) :3624 and groupByAndCount(Collection) :3666 (Document-typed).
    | AsyncMongoCollectionExecutor.java (absent; family at :3624-3666)
    | Medium
    | Add the two <T>…Class<T> overloads (mark @Beta) delegating to the sync typed methods. Same
      Document-only limitation as 5c.

(For symmetry the reverse direction was also checked: every async method has a sync counterpart, and the
async class adds nothing the sync class lacks except the wrapper-only sync()/AsyncExecutor plumbing — no
"async-only" orphan operations.)


=== 6. DEPRECATION ===
6a. mapReduce(String,String) and mapReduce(String,String,Class<T>) — already @Deprecated on both classes.
    | AsyncMongoCollectionExecutor.java:3693, :3721 (sync :4492, :4518)
    | Info (no action)
    | Correctly deprecated and correctly mirrored; @deprecated javadoc points to aggregate(...). No change.

No new deprecation candidates. (The gett* nullable family is an intentional, documented null-returning
counterpart to get*, not a deprecation candidate — consistent with the sync sibling.)


=== CROSS-CLASS NOTES ===
- Symmetry with MongoCollectionExecutor is high. Sync method count ~ async method count; the only
  structural asymmetries are the 8 missing async wrappers in sections 5a–5d (stream()/stream(Class);
  estimatedDocumentCount x2; groupBy(...,Class) x2; groupByAndCount(...,Class) x2) and the @Beta mismatch
  on the queryForXxx family (3a).
- Async return convention is CORRECT vs HBase: ContinuableFuture<T> wrapping, with Optional/Nullable/
  OptionalXxx kept INSIDE the future (HBase's AsyncHBaseExecutor uses the same ContinuableFuture idiom).
  Void sync ops -> ContinuableFuture<Void> via a Callable returning null (insertOne/insertMany), which is
  the standard pattern.
- No per-method Executor parameter anywhere (the AsyncExecutor is a construction-time field), so there is
  NO "Executor on some methods but not others" inconsistency — consistent by design.
- The reactivestreams twin (com.landawn.abacus.da.mongodb.reactivestreams) returns Publisher and is out of
  scope here; noted only conceptually.
