API DESIGN REVIEW — MongoCollectionExecutor (blocking per-collection executor)
File: abacus-da-all/src/main/java/com/landawn/abacus/da/mongodb/MongoCollectionExecutor.java
Reviewer model: opus-4.8   Date: 2026-06-23

=====================================================================
1. OVERALL ASSESSMENT
=====================================================================
This is a large (~4558-line), well-structured executor whose public surface is overwhelmingly
healthy: the exists/count/get/gett/findFirst/list/query/stream/queryForXxx families are internally
consistent, the Optional-vs-null contract is applied deliberately (Optional on get/findFirst,
null on gett, Nullable on queryForSingleValue, OptionalXxx on the primitive queryForXxx), pagination
(offset,count) and projection (Collection<String> selectPropNames vs Bson projection) are layered
uniformly, and the entity/Map/Document conversion rules are documented carefully (including the
$set-wrapping and null-property-dropping nuances). The write families (insert/update/replace/delete/
findOneAnd*/bulk*) faithfully wrap the driver and are by-design. The few actionable issues are
naming/return inconsistencies and a genuine overload-ambiguity footgun where the second argument of
update*/findOneAndUpdate is overloaded as either an Object update or a Collection pipeline (with
silently different server semantics). The class is otherwise correct and consistent.

=====================================================================
2. NAMING ISSUES
=====================================================================

[N1] mongoCollection() | MongoCollectionExecutor.java:211 | Low
  Accessor for the underlying MongoCollection is named mongoCollection() here, but the reactive twin
  (reactivestreams/MongoCollectionExecutor.java:216) names the same accessor coll(). Inconsistent
  accessor naming across the three mirrors. Recommendation: standardize one name across all three
  (mongoCollection() is the clearer/self-documenting choice; add coll() as an alias on the reactive
  twin or rename it). Confirming check: grep shows sync=mongoCollection(), async has no such accessor
  (only sync()), reactive=coll().

[N2] gett(...) family | MongoCollectionExecutor.java:612,639,667,696,726,758 | Low
  The "gett" name (double-t = "get and return null instead of Optional") is unusual and non-obvious
  to a first-time reader, but it is an established library-wide convention (mirrors the SQL/Dao
  executors' get/gett pair) and is documented. Keep as-is; flagged only for completeness. Confirming
  check: get(...) returns Optional<T>, gett(...) returns T-or-null — the pairing is consistent within
  this class.

[N3] groupBy/groupByAndCount return Stream of single-field {_id: value} | MongoCollectionExecutor.java:4232,4280
  | Low
  groupBy(String) is documented to emit {_id: <value>} documents (no projection to the field name in
  the Document.class path — see groupByPipeline at :4426 which returns only the $group stage when
  rowType==Document). The method name implies "group by field" but the Document result keys the value
  under "_id", not under the field name, whereas the typed (Class<T>) path projects it back to the
  field name. The Document vs typed result shapes differ. Recommendation: document this shape
  divergence on the no-Class overloads (or always $project _id back to the field name). Confirming
  check: groupByPipeline(String,...,Document.class) returns N.asList(new Document($group, {_id:$field})),
  while the typed path appends a $project mapping _id→field.

3. RETURN-TYPE AND PARAMETER ISSUES
=====================================================================

[R1] updateOne(Bson, Object) vs updateOne(Bson, Collection<?>)  [and updateMany, findOneAndUpdate]
  | MongoCollectionExecutor.java:2932 / 2992 ; 3171 / 3248 ; 3687 / 3799 | Medium
  The second argument is overloaded as either an Object (single update; wrapped in $set or passed
  through) or a Collection<?> (executed as an aggregation-PIPELINE update, which only permits
  $set/$addFields/$project/$unset/$replaceRoot/$replaceWith and REJECTS $inc/$push/etc.). The two
  paths have materially different and incompatible server semantics, yet are selected purely by the
  static type of the argument. A caller who passes a List-typed entity/update (e.g. a List built at
  runtime, or an entity that happens to be a Collection) silently gets pipeline semantics and a
  server error for ordinary operators. Recommendation: give the pipeline variant a distinct,
  self-documenting name (e.g. updateOnePipeline / updateManyPipeline / findOneAndUpdatePipeline) so
  the semantics are explicit at the call site rather than dispatched on Object-vs-Collection.
  Confirming check: toBson(Object) wraps/passes-through (:3055); toBson(Collection) builds List<Bson>
  passed to coll.updateOne(filter, List) (:3017-3026), which the driver runs as a pipeline.

[R2] insertOne returns void; bulkInsert returns int; insertMany returns void | MongoCollectionExecutor.java:2665,2735,3558
  | Low
  Within the insert family the return types are inconsistent: insertOne→void, insertMany→void,
  bulkInsert→int (inserted count). The driver's insert results (the generated _id / inserted count)
  are discarded for insertOne/insertMany but surfaced for bulkInsert. The reactive twin is more
  consistent here (insertOne→Mono<InsertOneResult>, insertMany→Mono<InsertManyResult>,
  bulkInsert→Mono<Integer>). Recommendation: consider returning the driver result (or at least an
  inserted count / generated id) from insertOne/insertMany for symmetry with bulkInsert and the
  reactive twin; at minimum document why they are void. Confirming check: coll.insertOne/insertMany
  return void here (:2700,2772) while coll.bulkWrite(...).getInsertedCount() is returned (:3593).

[R3] queryForSingleNonNull(String, Bson, Class) throws NPE on matched-but-null | MongoCollectionExecutor.java:1656
  | Low
  When a document matches but the field is absent/BSON-null (or conversion yields null), this returns
  Optional.of(null) which throws NPE (documented). This is a sharp edge: callers reaching for the
  "NonNull" variant to avoid null-handling get an exception instead of Optional.empty(). The Javadoc
  is explicit, but the behavior is still surprising for a method whose name suggests null-safety.
  Recommendation: collapse matched-but-null into Optional.empty() (use Optional.ofNullable) so the
  method's contract matches its name; or rename to make the throw-on-null behavior explicit.
  Confirming check: :1667 returns Optional.of(N.convert(getPropValueByPath(doc,...), valueType)) with
  no null guard; ground rule 5 (don't excuse a documented inconsistency).

[R4] queryForXxx primitive family collapses missing/null field to primitive default | MongoCollectionExecutor.java:1214-1456
  | Low
  queryForBoolean/Char/Byte/Short/Int/Long/Float/Double return a PRESENT OptionalXxx holding the
  primitive default (false/0/0.0) when a document matched but the field is absent/null, only returning
  empty() when NO document matched. This conflates "field present and equal to 0" with "field
  missing", and is a deliberate consequence of UNBOX. It is thoroughly documented per-method, and the
  escape hatch (queryForSingleValue with the boxed type) is named. Acceptable as-is given the @Beta
  marker and documentation; listed so the family is acknowledged rather than silently passed.
  Confirming check: each delegates to queryForSingleValue(...).mapToXxx(ToXxxFunction.UNBOX).

4. MISPLACED METHODS (target class)
=====================================================================
None. Every public method operates on the single wrapped collection (coll) and belongs on a
per-collection executor. watch()/aggregate()/distinct()/mapReduce()/groupBy* are all collection-scoped
and correctly placed here.

5. MISSING METHODS (target class)
=====================================================================

[M1] No typed-rowType overloads symmetric to other families on watch pipeline by Collection<String>
  | n/a | Low (non-actionable note)
  Not a real gap — watch already has Class<T> overloads. No action.

[M2] No find/list/stream "all" (no-filter) overloads except stream() | MongoCollectionExecutor.java:933,1701,2041
  | Low
  stream() and stream(Class) exist for the unfiltered "scan whole collection" case, but list/query
  have no no-filter overload (list(Bson) requires a non-null filter; same for query(Bson)). Callers
  must pass an explicit match-all filter (e.g. new Document()) to list/query the whole collection,
  whereas stream() offers the convenience. Recommendation: consider adding list(Class)/query(Class)
  (or list()/query()) unfiltered convenience overloads to match stream(). Confirming check: list(Bson)
  at :933 → list(filter, Document.class); executeQuery checks N.checkArgNotNull(filter) at :2502, so
  there is no unfiltered list/query path; only stream()/stream(Class) bypass the filter via coll.find().

=====================================================================
CROSS-CLASS NOTES (symmetry vs Async + reactive twins)
=====================================================================
- Async twin (AsyncMongoCollectionExecutor) is a near-1:1 mirror returning ContinuableFuture<...>.
  Naming, parameter order, and overload families match the sync class method-for-method EXCEPT:
    * Async is MISSING the typed groupBy(String,Class)/groupBy(Collection,Class)/
      groupByAndCount(String,Class)/groupByAndCount(Collection,Class) overloads that BOTH sync
      (:4255,4304,4358,4406) and reactive (:3850,3896,3958,4004) provide — async only exposes the
      Document-returning overloads (async :3561,3586,3624,3666). This is an async-class gap (out of
      scope for this report) but flagged for the async review.
    * Async names the queryForDate(String,Bson,Class) 3rd param "valueType" (async :1449) while sync
      (:1558) and reactive (:1626) name it "rowType". Trivial; pick one (valueType is more accurate
      since it is a single-value type, not a row type).
- Reactive twin (reactivestreams/MongoCollectionExecutor):
    * Exposes coll() instead of mongoCollection() — see [N1].
    * Has NO gett(...) family: get(...) returns Mono<T> (Mono is itself empty-or-present), so the
      Optional/null split collapses into one method set. Justified by reactive semantics; not a bug.
    * Primitive queryForXxx return Mono<Boolean>/Mono<Character>/... (boxed, empty-on-no-match) rather
      than OptionalXxx. Justified by reactive (no OptionalXxx-in-Mono). The empty-vs-default-on-null
      nuance therefore differs across twins — acceptable but worth noting for users porting code.
    * insertOne→Mono<InsertOneResult>, insertMany→Mono<InsertManyResult> (vs sync void) — see [R2];
      the reactive twin is the more consistent design and is a model for the sync return types.
- Return conventions overall: Optional on single-doc get/findFirst, null on gett, Nullable on
  queryForSingleValue (preserves matched-but-null), List on list, Dataset on query, Stream on
  stream/distinct/aggregate/groupBy*. This four-way (Optional/Nullable/null/empty) discipline is
  applied consistently within the sync class and is a strength.
