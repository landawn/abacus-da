1. OVERALL ASSESSMENT
AnyPut is a solid, thoroughly documented fluent wrapper over HBase `Put`, with the richest factory
surface of the four AnyMutation subclasses (Object/ByteBuffer keys, immutability and timestamp variants,
plus entity-based create(...) overloads). No correctness defects and no live deprecation candidates
(legacy addImmutable/setFamilyCellMap are already commented out). addColumn value semantics (overwrite
by type) are correct and well documented; the two adjacent byte[] family/qualifier params are inherent
to mirroring Put and shared by all siblings. Findings are cross-class consistency/parity items.
[RE-REVIEW 2026-06-24: Re-verified the full file (1556 lines) against live source. All findings below
re-confirmed; the only material code drift is that AnyPut's constructors now convert row keys via
HBaseExecutor.toRowBytes(...) (import AnyPut.java:18; e.g. ctor AnyPut.java:176) rather than
toRowKeyBytes — but toRowBytes/toRowKeyBytes/toValueBytes are still a single identity-delegating chain
(HBaseExecutor.java:2991-3007 -> toValueBytes 3026-3041, byte[] returned as-is at 3029-3030), so all
"byte[] is identity / cosmetic" reasoning still holds. The slice-param naming and factory-surface
findings are unchanged. No new correctness defects found.]

CROSS-CLASS NOTES
- Row-key factory surfaces are uneven across the four AnyMutation subclasses: AnyPut(rich Object/
  ByteBuffer/immutable/timestamp) vs AnyDelete(Object + familyMap) vs AnyIncrement/AnyAppend(byte[] +
  familyMap). Because toRowKeyBytes/toValueBytes treat byte[] as identity, the differences are largely
  cosmetic. Recommend a common minimum factory set with unified param names.
  [RE-REVIEW 2026-06-24: STILL-VALID. Helper-name note: AnyPut/AnyIncrement/AnyAppend now convert row
  keys via toRowBytes; AnyDelete via toRowKeyBytes (its familyMap ctor still via toRowBytes). All three
  helpers delegate to toValueBytes (HBaseExecutor.java:2991-3007), so the byte[]-identity argument is
  unaffected by the rename.]
- add(Cell): AnyPut/AnyDelete/AnyIncrement declare `throws IOException`, AnyAppend does not — faithful
  mirror of HBase's own Mutation-subclass differences, NOT an AnyPut defect.
  [RE-REVIEW 2026-06-24: STILL-VALID — AnyPut.add(Cell) throws IOException at AnyPut.java:1369;
  AnyAppend.add(Cell) declares no checked exception at AnyAppend.java:604.]
- equals/hashCode identity-based on the wrapped Put — consistent with AnyPut's documentation.
  [RE-REVIEW 2026-06-24: STILL-VALID — hashCode AnyPut.java:1395-1398, equals AnyPut.java:1429-1441
  delegate to Put.equals/hashCode; Javadoc documents identity semantics.]
